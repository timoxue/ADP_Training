    方案A：话术合规预检助手（第五步详细操作）¶

    单工作流模式 | DocParse → 知识库检索 → 标签提取 → 条件判断 → 风险清单 | app_id: 2068701020736650752

    六步推理链导览

    本文档是第五步：分步操作的完整配置指南。第一步到第四步的分析答案见 场景三六步分析。

    培训目标¶

    完成本培训后，你将能够：

    1. 在ADP平台上搭建单文件合规预检子工作流（含DocParse解析 + 条件分支：高/中/低风险分流）
    2. 在ADP平台上搭建多文件批量合规预检主工作流（使用ITERATION迭代节点）
    3. 理解LOGIC_EVALUATOR条件判断的配置方法和分支路由原理
    4. 独立调试和排查工作流问题

    整体架构¶

    核心设计：文档解析 → 知识库检索 → 标签提取 → 条件判断 → 风险清单¶

    ┌──────────────────────────────────────────────────────────────────────────┐
│  主工作流 compliance_precheck（多文件批量预检）                              │
│                                                                          │
│  [开始] → [文件收集(5个)] → [逐文件预检(ITERATION)] → [汇总输出] → [结束]    │
│                                         │                                │
│                                         │ 每个文件迭代一次                  │
│                                         ▼                                │
│  ┌───────────────────────────────────────────────────────────────────┐   │
│  │  子工作流 compliance_single_check（单文件预检 + 条件分支）            │   │
│  │                                                                   │   │
│  │  [开始] → [文档解析] → [知识库检索] → [合同风险分析] → [标签提取]    │   │
│  │                                                        │           │   │
│  │                                             [条件判断(LOGIC_EVALUATOR)]  │   │
│  │                                              /         |         \  │   │
│  │                                        高风险      中风险      低风险│   │
│  │                                          ↓           ↓           ↓  │   │
│  │                                     [拦截输出]  [警告输出]  [通过输出]│   │
│  │                                          \           |           /  │   │

│  └───────────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────────┘

    工作原理¶

    1. 用户上传多个文件（最多5个）
    2. FILE_COLLECTION收集所有文件，输出FileList数组
    3. ITERATION节点遍历FileList，每次取1个文件传给子工作流
    4. 子工作流对每个文件执行：
    5. 文档解析（DocParse解析文件URL，输出文件文本内容）
    6. 知识库检索（用文件内容检索相关法规条文和禁语词表，为LLM提供真实法条依据）
    7. LLM合规分析（结合知识库检索结果和文件内容，检出违规、评估风险等级）
    8. PARAM_EXT标签提取（结构化提取7个参数）
    9. LOGIC_EVALUATOR条件判断（按risk_level分流）
    10. 高风险 → 拦截输出（阻断+必须修改）
    11. 中风险 → 警告输出（有条件通过+建议）
    12. 低风险 → 通过输出（通过+优化建议）
    13. 主工作流ANSWER汇总所有迭代结果，输出批量预检报告

    Phase 1：搭建子工作流 compliance_single_check¶

    为什么先建子工作流

    子工作流是单文件预检的核心流程，每次处理1个文件，含条件分支。必须先建子工作流，因为主工作流的ITERATION节点需要引用它。

    步骤1：创建子工作流¶

    操作路径：ADP控制台 → 应用管理 → 合规预检助手 → 工作流管理 → 新建工作流

    字段 填写内容
    工作流名称 compliance_single_check
    描述 逐文件合规预检-子工作流（条件分支）

    点击「确定」→ 进入画布编辑器。

    步骤2：搭建10节点分支流¶

    从左侧节点面板依次拖入10个节点：

    [开始] → [文档解析] → [知识库检索] → [合同风险分析] → [标签提取] → [条件判断]
                                                                    │
                                                        ┌───────────┼───────────┐
                                                        ▼           ▼           ▼
                                                    [拦截输出]  [警告输出]  [通过输出]
                                                        └───────────┼───────────┘
                                                                    ▼
                                                                  [结束]

    主干连线（从节点右侧拖到下一节点左侧）：

    1. [开始] → [文档解析]
    2. [文档解析] → [知识库检索]
    3. [知识库检索] → [合同风险分析]
    4. [合同风险分析] → [标签提取]
    5. [标签提取] → [条件判断]

    分支连线（从条件判断节点拖出3条线）：

    1. [条件判断] → [拦截输出]（高风险分支）
    2. [条件判断] → [警告输出]（中风险分支）
    3. [条件判断] → [通过输出]（低风险分支）

    汇合连线：

    1. [拦截输出] → [结束]
    2. [警告输出] → [结束]
    3. [通过输出] → [结束]

    连线注意

    条件判断节点拖出3条线时，每条线对应一个分支条件。在LOGIC_EVALUATOR配置中需要为每条线设置条件表达式。

    步骤3：配置「开始」节点¶

    点击「开始」节点 → 右侧配置面板 → 找到「工作流参数」区域。

    添加2个工作流参数（这些参数由主工作流的ITERATION节点映射传入）：

    # 参数名 类型 描述 必填
    1 file_url STRING 当前处理的文件URL 是
    2 file_name STRING 当前处理的文件名 否

    操作方式：

    1. 点击「添加参数」按钮
    2. 参数名输入 file_url，类型选 STRING，描述填"当前处理的文件URL"，勾选必填
    3. 重复步骤添加 file_name，描述填"当前处理的文件名"

    参数名必须一致

    这2个参数的名称必须和主工作流ITERATION的输入映射完全一致，否则数据传不进来。ITERATION每次迭代传入 file_list.Item.FileURL → file_url，file_list.Item.FileName → file_name。

    步骤4：配置「文档解析(PLUGIN)」节点¶

    这个节点把ITERATION传入的文件URL解析为文本内容，让下游的知识库检索和LLM都能读取文件内容。

    4.1 选择插件

    点击「文档解析」节点 → 右侧配置面板 → 点击「选择插件」→ 搜索 DocParse → 选择该插件。

    配置项 值
    插件名称 DocParse
    ToolID 44b4c776-b1a4-4b3a-a2dc-7355c1ad193b
    支持格式 PDF, DOC, DOCX, PPT, PPTX, TXT, XLSX, CSV

    4.2 输入变量

    添加1个输入变量：

    变量名 引用类型 引用节点 引用字段 说明
    file_url 引用节点输出 开始 file_url ITERATION传入的文件URL

    操作方式：

    1. 点击「添加变量」
    2. 变量名输入 file_url
    3. 引用类型选择「引用节点输出」
    4. 引用节点选择「开始」
    5. 引用字段选择 file_url

    4.3 输出说明

    DocParse解析文件后输出3个字段：

    输出字段 类型 说明
    Output.Code INT 返回码。0=正常，非0=异常
    Output.Msg STRING 返回信息
    Output.Data.Answer STRING 文件解析出的Markdown文本 ← 文件内容在这里

    为什么用DocParse

    DocParse把文件URL解析为可阅读的文本内容。下游的知识库检索和LLM都引用 Output.Data.Answer 获取文件内容，这样两个节点看到的是同一份文件文本，分析更准确。

    步骤5：配置「知识库检索(KNOWLEDGE_RETRIEVER)」节点¶

    这个节点用DocParse解析出的文件内容检索知识库中的法规条文和禁语词表，为LLM提供准确的法规依据，避免LLM凭记忆编造法条。

    5.1 基础配置

    知识库检索节点不需要配置模型，它只做向量检索，不涉及大模型生成。配置项如下：

    配置项 填写值 说明
    节点名称 知识库检索 —
    用户问题 {{query}} 引用输入变量query（文件内容），用于检索相关法规

    5.2 知识库选择

    在「知识检索」配置区域：

    配置项 选择 说明
    知识 全部知识库 检索应用关联的所有知识库
    筛选方式 全部知识 检索所有文档
    文档 ✅ 开启 检索文档类知识（法规汇编、禁语词表等）
    问答 ✅ 开启 检索QA问答对（如有）
    数据库 ❌ 关闭 不需要数据库检索

    确保以下4个分类的文档已上传且状态为"导入完成"：

    分类 内容 作用
    合规规则库 《证券法》《金融营销通知》等法规条文 提供法规依据
    禁语词表 68个禁用词清单 精确匹配违规词
    合同模板 标准合同模板含标注 合同条款比对
    历史预检案例 通过/不通过案例 参考判例

    5.3 输入变量

    添加1个输入变量（用文件内容作为检索query）：

    变量名 引用类型 引用节点 引用字段 说明
    query 引用节点输出 文档解析 Output.Data.Answer 用文件内容检索相关法规

    操作方式：

    1. 点击「添加变量」
    2. 变量名输入 query
    3. 引用类型选择「引用节点输出」
    4. 引用节点选择「文档解析」
    5. 引用字段选择 Output.Data.Answer

    5.4 输出说明

    知识库检索节点输出多个字段，关键字段如下：

    输出路径 类型 说明
    Output.Content STRING 检索到的内容（QA答案或DOC文档片段）← LLM引用这个
    Output.KnowledgeList Array 检索返回的数据列表
    Output.KnowledgeType STRING 检索片段类型（QA问答对/DOC文档）
    Output.DocName STRING 文档名（DOC类型时有效）
    Output.Confidence NUMBER 切片的置信度

    输出字段注意

    LLM引用的是 Output.Content，不是 Output.Data.Answer。知识库检索节点和DocParse节点的输出字段路径不同，配置时请注意区分。

    步骤6：配置「合同风险分析(LLM)」节点¶

    这是核心分析节点，接收文件内容（来自DocParse）和知识库检索结果后执行合规预检。

    6.1 基础配置

    参数 填写值 原因
    节点名称 合同风险分析 —
    模型(ModelName) Deepseek/deepseek-v3.2 分析能力强，适合合规分析
    Temperature 0.1 合规场景需要确定性输出，低温度减少随机性

    你是「合规预检助手」，专门为证券行业营销话术和合同文档提供合规预检服务。你的职责是检出违规表述、评估风险等级、给出合规建议。你必须严格遵守三条红线：不承诺收益、不代客操作、不构成投资建议。

    LLM Prompt¶

    ## 三条不可逾越的红线
1. ❌ 不承诺收益 — 不得出现保本/保收益/稳赚/无风险等表述
2. ❌ 不代客操作 — 不得出现帮你买入/替你操作/代你交易等表述
3. ❌ 不构成投资建议 — 不得出现强烈推荐/必买/必涨/闭眼买等表述

## 预检流程
1. 判断输入类型：marketing(营销话术) / contract(合同) / other(其他)
2. 营销话术：逐项检查是否命中以下6类违规：
   - 保本承诺（保本/保收益/稳赚/无风险/零风险/保证盈利）
   - 代客操作（帮你买入/替你操作/代你交易/我来操盘）
   - 投资建议诱导（强烈推荐/必买/必涨/闭眼买/赶紧加仓）
   - 误导性表述（高收益/秒到账/零门槛/内部消息/庄家拉升）
   - 不当背书（证监会推荐/监管背书/政府扶持）
   - 业绩展示不规范（年化X%无来源/历史最佳业绩）
3. 合同：检查是否包含以下关键条款：
   - 风险提示条款（是否充分披露风险）
   - 适当性评估条款（是否匹配投资者风险等级）
   - 代客操作禁止条款（是否明确禁止代客操作）
   - 收益禁止承诺条款（是否明确禁止承诺收益）
   - 如含违规条款（如隐含收益承诺），标记为高风险
4. 如为"other"或超出预检范围（如用户要求投资建议），标记is_redline=true，risk_level=high

## 风险等级判定标准
- high（高风险）：命中红线词（保本/保收益/稳赚/代客操作/投资建议诱导），或命中2个以上高风险词
- medium（中风险）：命中1个高风险词但非红线，或命中2个以上中风险词（误导性表述/不当背书）
- low（低风险）：仅命中低风险词（模糊表述/措辞建议），或无违规项

    为什么用条件分支¶

    维度 说明
    处置精度 高/中/低风险分别用不同模板，不同处置语气
    高风险处置 🚫拦截+明确阻断提示+必须修改后重新提交
    中风险处置 ⚠️警告+有条件通过+需合规部门复核
    低风险处置 ✅通过+优化建议为可选
    可读性 通过emoji标识一目了然
    合规要求 检出（PARAM_EXT）和处置（分支ANSWER）分离

    LOGIC_EVALUATOR条件分支的工作原理¶

    PARAM_EXT输出 risk_level = "high"
        ↓
LOGIC_EVALUATOR收到 risk_level
        ↓
逐个匹配条件：
  分支1: risk_level EQ "high"   → ✅ 匹配！ → 路由到「拦截输出」
  分支2: risk_level EQ "medium" → ❌ 不匹配
  分支3: risk_level EQ "low"    → ❌ 不匹配
        ↓
只有「拦截输出」节点被执行
        ↓
「拦截输出」→ 「结束」

    LOGIC_EVALUATOR是纯流程控制节点，不产生数据输出。它的作用是根据条件值选择走哪条分支。

    三条红线与风险等级的对应关系¶

    红线 触发条件 risk_level 走哪个分支
    ❌ 不承诺收益 出现保本/保收益/稳赚/无风险等 high 🚫 拦截输出
    ❌ 不代客操作 出现帮你买入/替你操作/代你交易等 high 🚫 拦截输出
    ❌ 不构成投资建议 出现强烈推荐/必买/必涨/闭眼买等 high 🚫 拦截输出
    — 命中1个高风险词但非红线 medium ⚠️ 警告输出
    — 仅命中低风险词或无违规 low ✅ 通过输出

    附录D：文件解析方案选择¶

    子工作流含 DocParse 节点，直接在迭代内解析文件内容。如评估其他方案，可参考以下对比：

    方案 迭代传参方式 优点 缺点 适用场景
    方案A — 子工作流含DocParse（本文档采用） file_url → file_list.Item.FileURL 完整解析文件内容，LLM基于真实文件文本分析 调试时需上传文件 生产环境，需解析文件内容
    方案B — 用户手动输入 content → USER_INPUT，file_name → FileName 调试不需要上传文件，直接输入文本 上传文件时需同时手动粘贴内容 开发调试、培训演示
