# 操作步骤详细清单

本文件是第五步「分步操作」的详细参考，每个步骤包含具体操作、注意事项和常见问题。

---

## 5A 手工操作分解

手工操作分解的目标是让用户**理解每一步做什么、为什么这样做**，而不是盲目照做。展示完手工分解后，再询问用户是否需要自动搭建。

### 按模式分支

不同模式的操作步骤有显著差异。根据第三步选择的应用模式，展示对应的操作清单：

**Claw 模式**（8步，⚠️ 无合规硬拦截设置项）：
1. 创建应用（选Claw模式）→ 2. 设计提示词（框架B）→ 3. 配置知识库 → 4. 挂载Skills → 5. 挂载工具 → 6. 配置欢迎语 → 7. 测试调优 → 8. 发布上线
> ⚠️ Claw 模式没有"配置平台设置"步骤——应用设置仅有欢迎语/对话体验/变量/高级设置(同义词)4项，没有内容安全/保守回答/隐私脱敏/长期记忆/上下文轮数。合规风控只能通过提示词软约束实现。

**标准模式**（10步）：
1. 创建应用（选标准模式）→ 2. 配置模型 → 3. 设计提示词（框架A）→ 4. 配置知识库 → 5. 配置连接器与工具 → 6. （可选）配置工作流 → 7. 配置平台设置 → 8. 配置欢迎语 → 9. 测试调优 → 10. 发布上线

**单工作流模式**（10步）：
1. 创建应用（选单工作流模式）→ 2. 配置模型 → 3. 设计提示词（简短入口引导）→ 4. 配置知识库 → 5. 配置连接器与工具 → 6. 编排工作流（核心步骤）→ 7. 配置平台设置 → 8. 配置欢迎语 → 9. 测试调优 → 10. 发布上线
> 单工作流核心差异：所有业务逻辑都在工作流中定义；提示词只需做入口引导；知识库在工作流节点中使用（非对话直接检索）；步骤6编排工作流是最核心最耗时的步骤

**Multi-Agent模式**（11步）：
1. 创建应用（选Multi-Agent模式）→ 2. 配置模型 → 3. 定义子Agent结构 → 4. 编写每个Agent提示词 → 5. 配置转交描述 → 6. 配置知识库 → 7. 挂载Skills与工具 → 8. 配置平台设置 → 9. 配置欢迎语 → 10. 测试调优 → 11. 发布上线
> Multi-Agent 核心差异：需要定义调度Agent+子Agent的分工和转交逻辑；提示词拆为多段；知识库问答需手动启用工具；测试重点在调度准确性

## 步骤 5.1：创建应用

**使用技能**：`adp-app-manager` → `create_app`

**前置条件**：
- 已明确应用名称和描述（来自第一步业务产出物）
- 环境变量 `ADP_API_KEY` 已设置

**操作流程**：
1. 确认应用名称、描述
2. 调用 `create_app`，传入 `name`、`description`、`space_id`
3. 保存返回的 `app_id`，后续所有操作都基于此 ID

**自动搭建时必须二次确认**——执行 `create_app` 前，向用户展示确认信息：
```
📋 即将创建应用，请确认：

应用名称：[XXX]
应用模式：[Claw/标准/单工作流/Multi-Agent]
核心配置摘要：
  - 提示词：[一句话概括核心逻辑]
  - 知识库：[N个分类/N份文档]
  - Skills：[N个，核心技能：XXX]
  - 工具：[N个，核心工具：XXX]

⚠️ [模式专属提醒，如"Claw模式创建后不可切换"]

确认创建？(是/否)
```
用户明确回复"是/确认/OK"后才执行。模糊回复不算确认。

**注意事项**：
- `create_app` 是高风险操作，需二次确认（风险提示+改动摘要+确认问句）
- 创建后 `app_id` 自动沉淀为本会话首选 ID
- 如已有应用，跳过此步，直接用 `describe_app` 确认当前状态

## 步骤 5.2：设计提示词

**使用技能**：`adp-prompt-builder`

**前置条件**：
- 已完成第一至四步（交付物、能力、映射、推理结论清晰）

**操作流程**：
1. 根据第二步的能力拆解决定提示词框架：
   - 标准模式 → 框架A（角色+风格+输出模块+限制）
   - Agent模式 → 框架B（任务目标+任务流程+限制）
2. 逐项填写框架中的占位符
3. 把设计推理中的边界和兜底写入限制部分
4. 生成完整提示词文本

**提示词设计要点**：
- 输出模块要对应交付物形态（如投研助手的"关键指标对照表"对应输出模块）
- 限制部分要对应合规风控需求（如财富顾问助手的"四条红线"）
- 任务流程要对应能力拆解（如投研助手的"知识库检索→API查询→交叉校验→输出"）

## 步骤 5.3：配置知识库

**使用技能**：`tencentcloud-adp-kb-management`

**前置条件**：
- 已确定知识库分类结构（来自第三步功能映射）
- 已准备好文档文件

**操作流程**：
1. 创建分类树：`create_doc_cate.py --bot-biz-id {app_id} --parent {parent_id} --name {name}`
2. 上传文档到对应分类：`save_doc.py --bot-biz-id {app_id} --file {path} --cat-biz-id {cate_id}`
3. 等待文档解析完成（状态流转：解析中→审核中→学习中→导入完成）
4. 用 `list_doc.py --bot-biz-id {app_id}` 确认所有文档状态为"导入完成"

**注意事项**：
- 文档解析可能需要几分钟，大量文档需等待
- PDF 表格可能丢结构，验收时需重点检查
- FAQ 导入不支持 MD 格式，需转 CSV/Excel
- 检索配置建议：混合检索、Top-K 5-10、相似度 0.5-0.7、必须开引用溯源
- 前置依赖：`pip install cos-python-sdk-v5 --break-system-packages -q`

## 步骤 5.4：挂载技能与工具

**使用技能**：`adp-app-manager` → `modify_agent_skill_list` / `modify_agent_tool_list`

**前置条件**：
- 已确定需要的技能列表（来自第三步功能映射）
- 已有 `app_id`

**操作流程**：
1. 先调 `describe_agent_config` 获取当前 `skill_list` 和 `tool_list`
2. 调 `describe_skill_list` 获取可用技能列表，找到目标技能的 `skill_id`
3. 调 `describe_plugin_list` 获取可用工具列表，找到目标工具的 `plugin_id` + `tool_id`
4. 调 `modify_agent_skill_list` 挂载技能，`add_skill_list` 格式为 `["id1", "id2"]`（纯字符串数组，不是对象数组）
5. 调 `modify_agent_tool_list` 挂载工具

**注意事项**：
- **必须用真实 ID，不能用显示名称代替**（如不能用 `"pdf"` 代替 `"xxx-xxx-xxx"`）
- `add_skill_list` 是字符串数组 `["id1", "id2"]`，不是 `[{"skill_id": "id1"}]`
- 修改前必须先读当前值，避免重复挂载
- 展示变更摘要给用户确认后再执行
- 批量操作间隔 3 秒以上，避免 403 限流

## 步骤 5.4a：定义子Agent结构（仅 Multi-Agent 模式）

**使用技能**：`public-adp-multiagent-creator`

**前置条件**：
- 已完成第一至四步（交付物、能力、映射、推理结论清晰）
- 已确定需要的子Agent数量和分工（来自第三步功能映射）

**操作流程**：
1. 规划子Agent结构——调度Agent + N个子Agent（建议不超过5个）
2. 为每个子Agent定义：
   - 名称：简短明确（如"财务分析师""舆情专员"）
   - 职责：一句话描述本Agent负责什么
   - 专用技能：从第三步映射结果中选取
   - 输出格式：标准化模板（方便调度Agent汇总）
   - 转交描述：什么条件下调度Agent会把问题转交给本Agent
3. 为调度Agent定义：
   - 转交规则：什么问题转给哪个子Agent
   - 并行策略：哪些子Agent可以同时执行
   - 汇总逻辑：如何整合多个子Agent的输出

**子Agent结构规划模板**：

```
调度Agent：[名称]
  职责：接收用户问题→判断类型→分发给子Agent→汇总返回
  转交规则：
    - 涉及[能力1] → 转交 [子Agent1]
    - 涉及[能力2] → 转交 [子Agent2]
    - 涉及[能力3] → 转交 [子Agent3]
    - 多项并行 → 同时转交多个子Agent
  汇总模板：
    【模块1】[子Agent1名称]分析：[输出摘要]
    【模块2】[子Agent2名称]分析：[输出摘要]
    ...

子Agent1：[名称]
  职责：[具体做什么]
  专用技能：[skill_id列表]
  输出格式：
    ```
    ## [子Agent1名称]分析报告
    ### 1. [核心发现]
    ### 2. [关键数据]
    ### 3. [风险/建议]
    ```
  转交描述：当用户需要[具体能力]时转交

子Agent2：[名称] ...
```

**注意事项**：
- 子Agent数量建议不超过5个——超过5个调度复杂度指数增长，容易转交混乱
- 转交描述必须互斥——两个子Agent的转交描述重叠会导致调度Agent判断混乱
- 输出格式必须标准化——调度Agent需要解析子Agent输出做汇总，格式不统一会丢信息
- 技能挂载是应用级别——所有子Agent共享所有技能，无法限制某个子Agent只用特定技能（通过提示词约束）

## 步骤 5.4b：编写每个Agent的提示词（仅 Multi-Agent 模式）

**使用技能**：`adp-prompt-builder`

**前置条件**：
- 已完成步骤 5.4a（子Agent结构规划完毕）

**操作流程**：
1. 调度Agent提示词（框架B-Agent模式）：
   - 任务目标：接收问题→分析→分发→汇总
   - 转交规则：明确什么问题转给谁（必须与转交描述一致）
   - 汇总模板：最终输出给用户的格式
   - 限制：不做专业分析，只做调度和汇总
2. 每个子Agent提示词（框架B-Agent模式）：
   - 任务目标：本Agent负责的专项分析
   - 输入规范：期望收到什么信息
   - 输出格式：严格按步骤5.4a定义的模板
   - 边界限制：超出本领域的问题不回答，转回调度Agent

**调度Agent提示词关键要素**：

```markdown
# 任务目标
你是「[应用名]」的调度Agent，负责接收用户问题、分析需求、分发给专业子Agent、汇总返回。

# 转交规则
- 涉及[能力1]（如：XXX）→ 转交 [子Agent1]
- 涉及[能力2]（如：XXX）→ 转交 [子Agent2]
- 涉及[能力3]（如：XXX）→ 转交 [子Agent3]
- 同时涉及多项 → 并行转交，全部返回后汇总
- 不在任何子Agent范围内 → 直接回答并提示能力边界

# 汇总模板
[定义最终输出格式]

# 限制
- 你不做专业分析，只做调度和汇总
- 子Agent返回的信息你直接引用，不二次加工
```

**子Agent提示词关键要素**：

```markdown
# 任务目标
你是「[子Agent名]」，负责[具体职责]。

# 输出格式（严格遵守）
[标准化模板]

# 可用技能
- [skill1]：[何时调用]
- [skill2]：[何时调用]

# 限制
- 只回答本领域问题，超出范围转回调度Agent
- 输出必须按上述格式，不要自创格式
```

**注意事项**：
- 调度Agent的转交规则必须与子Agent的转交描述完全一致——不一致会导致转交失败
- 子Agent输出格式越标准化，调度Agent汇总越可靠
- 可用 `public-adp-multiagent-creator` 技能辅助生成 Multi-Agent 框架

## 步骤 5.4c：配置转交描述（仅 Multi-Agent 模式）

**使用技能**：`adp-app-manager`

**前置条件**：
- 已完成步骤 5.4a 和 5.4b

**操作流程**：
1. 在 ADP 控制台打开 Multi-Agent 应用
2. 进入每个子Agent的编辑页
3. 在「转交描述」字段填入步骤 5.4a 规划的转交描述
4. 确认所有子Agent的转交描述互不重叠

**转交描述写法**：
- 一句话描述"什么情况下调度Agent应该把问题转交给这个子Agent"
- 包含关键词触发条件
- 示例："当用户需要财务数据提取、年报分析、财务指标计算、DCF估值时转交"

**注意事项**：
- **转交描述必须互斥**——如果两个子Agent的描述都包含"分析"，调度Agent会混乱
- 解决方法：每个子Agent的转交描述使用领域专属关键词（如"财务指标"vs"行业对比"vs"新闻舆情"vs"风险预警"）
- 转交描述不宜过长——调度Agent用关键词匹配，写太多反而稀释匹配度## 步骤 5.5：配置平台设置（按模式区分）

**使用技能**：`adp-app-manager` → `describe_app` 确认配置

### ⚠️ 关键：Claw 模式与标准模式的设置项完全不同

**Claw 模式应用设置**（菜单路径：右上角「应用设置」）：

| 设置项 | 是否存在 | 说明 |
|---|---|---|
| 欢迎语 | ✅ | 可配置欢迎语和开场问题 |
| 对话体验 | ✅ | 含聊天背景设置 |
| 变量 | ✅ | API 传参变量（如 {{API.CustomerName}}） |
| 高级设置 | ✅ | 含同义词配置 |
| 内容安全 | ❌ **不存在** | Claw 模式无此设置项 |
| 保守回答 | ❌ **不存在** | Claw 模式无此设置项 |
| 隐私脱敏 | ❌ **不存在** | Claw 模式无此设置项 |
| 长期记忆 | ❌ **不存在** | Claw 模式无此设置项 |
| 上下文轮数 | ❌ **不存在** | Claw 模式模型参数通过 API 设置（max_tokens/temperature） |
| 模型设置 | ⚠️ 有限 | 单模型体系（默认Kimi K2.5可切换），非双模型协同 |

**标准/单工作流/Multi-Agent 模式应用设置**（菜单路径：右上角「应用设置」）：

| 设置项 | 是否存在 | 操作方式 |
|---|---|---|
| 内容安全 | ✅ | 用户手动开启 |
| 保守回答 | ✅ | 应用设置→回复设置→保守回答，用户手动开启 |
| 隐私脱敏 | ✅ | 应用设置→隐私保护，用户手动开启 |
| 长期记忆 | ✅ | 应用设置→长期记忆，用户手动开启 |
| 上下文轮数 | ✅ | 应用设置→模型设置，建议 10-20 轮 |
| 模型设置 | ✅ | 思考模型+生成模型双模型体系 |

**关键提醒**：
- **不要对 Claw 模式用户说"去开启内容安全/保守回答/隐私脱敏"——这些设置项在 Claw 模式下不存在**
- Claw 模式的合规风控只能通过提示词约束实现（软约束），没有平台级硬拦截
- 如果场景需要内容安全/保守回答/隐私脱敏的硬拦截，应建议用户选择标准模式
- 标准模式下长期记忆开关**不在 Agent 编辑页**，在「应用设置」独立菜单项

## 步骤 5.6：配置欢迎语与开场问题

**使用技能**：`adp-app-manager` → `modify_app_greeting`

**操作流程**：
1. 先调 `describe_app` 获取当前 `greeting` 和 `opening_question_list`
2. 设计欢迎语文本和开场问题（最多 3 条）
3. 调用 `modify_app_greeting` 提交修改
4. 只改开场问题时不要回填原 `greeting`

**注意事项**：
- **开场问题最多 3 条**，超过会被平台拒绝
- 只传需要修改的字段，不要回填不变的字段（回填会让平台把未变字段也算进"本次修改"）
- 展示变更给用户确认后再执行

## 步骤 5.7：配置工作流（可选）

**使用技能**：`adp-workflow-package-generator`

**适用条件**：应用需要多步骤固定流程（如：收集信息→查询→分析→生成报告）

**操作流程**：
1. 检查当前 Agent 是否已安装 `adp-workflow-package-generator` 技能
2. 如已安装，按该技能的流程生成工作流包
3. 调用 `import_workflow` 导入到应用
4. 如未安装，提示用户先安装该技能

**注意事项**：
- 不要自行手写 workflow JSON，必须走脚本生成
- 生成后需运行 `verify_delivery.py` 校验
- `import_workflow` 是高风险操作，需二次确认

## 步骤 5.8：配置连接器（可选）

**使用技能**：`adp-plugin-manager`

**适用条件**：需要对接第三方平台（腾讯文档、GitHub、QQ邮箱等）

**操作流程**：
1. 搜索可用连接器：`python3 scripts/plugin_manager.py search "平台名" --limit 10`
2. 推荐合适的连接器给用户
3. 引导用户在 Agent 配置中启用该连接器
4. 用户确认已启用后，继续后续任务

**注意事项**：
- 本技能只搜索和引导启用，不直接执行连接器操作
- 搜索结果不代表已启用，需要用户手动启用
- 已有可直接调用的工具时，不触发此技能

---

## 步骤 5.9：提交提示词

**使用技能**：`adp-app-manager` → `modify_agent_prompt`

**操作流程**：
1. 先调 `describe_agent_config` 获取当前提示词（`agent.instructions` 字段）
2. 与新提示词做 diff，展示修改前/修改后/变更摘要
3. 用户确认后，调用 `modify_agent_prompt` 提交
4. 入参 `system_prompt` 不支持传空字符串清空

**注意事项**：
- **必须先读再写**：平台是整段替换语义，不读直接写会覆盖已有配置
- `system_prompt` 不支持清空，客户要求清空时改为"最简一句话版本"
- 长文本务必用 `-f path.json` 或 `-f -` 传参，避免 `ARG_MAX` 限制
- 展示 diff 后等用户确认，分轮执行

---

## 步骤 5.10：发布上线

**使用技能**：`adp-app-manager` → `release_app`

**操作流程**：
1. 回顾本会话是否已做过 `modify_*` 操作
2. 展示本次所有变更摘要
3. 二次确认（风险提示+改动摘要+确认问句）
4. 用户明确肯定后，调用 `release_app`

**注意事项**：
- 高风险操作，需严格二次确认
- 模糊回复（"看下""我想想"）不算确认，继续追问
- 发布后修改需再次发布才能生效

## 5B 自动搭建

**⚠️ 关键流程：先完整展示 5A 手工分解 → 用户确认理解 → 再询问搭建方式**

不能在展示手工分解的过程中就问"要不要我帮你搭"——用户还没看完所有步骤，无法做出知情决策。

### 5B.1 询问时机

1. 展示对应模式的完整操作清单（5A）
2. 等用户确认"看完了/理解了/OK"
3. 再询问搭建方式选择

### 5B.2 搭建方式选项（三选一）

| 方式 | 说明 | 适合谁 |
|---|---|---|
| **手工搭建** | 用户自己在 ADP 控制台按步骤操作 | 想完全掌控每步、学习操作流程的用户 |
| **半自动搭建** | 我完成可自动化的步骤（创建应用、挂载Skills、提交提示词等），用户手动完成平台设置（内容安全/保守回答/隐私脱敏/长期记忆等需在控制台开启的项） | 想省时间但也想亲手做关键设置的用户 |
| **全自动搭建** | 我完成所有可自动化的步骤，最后列出用户需手动完成的项 | 想最快交付、理解了原理但不想动手的用户 |

**话术**：

> 「以上是完整的操作步骤分解。你现在可以选择搭建方式：
> 1. **手工搭建**——在 ADP 控制台按步骤操作，每步自己把控
> 2. **半自动搭建**——我来完成创建应用、提示词、知识库分类、挂载 Skills/工具、欢迎语等，你手动开启平台设置（内容安全/保守回答等）
> 3. **全自动搭建**——我完成所有可自动化的步骤，最后给你一份手动操作清单
>
> 无论哪种方式，自动化的每一步都会先展示变更摘要，你确认后再执行，不会偷偷改配置。」

### 5B.3 自动搭建步骤与技能对应

| 步骤 | 调用技能 | 具体操作 | 适用模式 | 需要确认 |
|---|---|---|---|---|
| 创建应用 | `adp-app-manager` | `create_app`（需二次确认） | 全部 | ✅ 高风险，必须二次确认 |
| 设计提示词 | `adp-prompt-builder` + `adp-app-manager` | 先用 prompt-builder 生成，再用 `modify_agent_prompt` 提交 | 全部 | ✅ 展示 diff 后确认 |
| Multi-Agent 子Agent提示词 | `adp-prompt-builder` + `public-adp-multiagent-creator` | 逐个生成子Agent提示词+转交描述 | Multi-Agent | ✅ 逐个展示后确认 |
| 配置知识库 | `tencentcloud-adp-kb-management` | 创建分类 → 上传文档 | 全部 | ✅ 展示分类结构后确认 |
| 挂载 Skills | `adp-app-manager` | `modify_agent_skill_list` | 全部 | ✅ 展示变更摘要后确认 |
| 挂载工具 | `adp-app-manager` | `modify_agent_tool_list` | 全部 | ✅ 展示变更摘要后确认 |
| 配置欢迎语 | `adp-app-manager` | `modify_app_greeting` | 全部 | ✅ 展示内容后确认 |
| 配置工作流 | `adp-workflow-package-generator` + `adp-app-manager` | 生成包 → `import_workflow` | 标准/单工作流 | ✅ 高风险，必须二次确认 |
| 配置连接器 | `adp-plugin-manager` | 搜索推荐 → 引导启用 | 全部 | ℹ️ 仅引导，用户自行启用 |
| 平台设置 | ❌ 无法自动 | 内容安全/保守回答/隐私脱敏/上下文轮数 | 标准/单工作流/Multi-Agent | ℹ️ 需在控制台手动开启 |
| 发布上线 | `adp-app-manager` | `release_app` | 全部 | ✅ 高风险，必须二次确认 |

### 5B.4 自动搭建的安全规则

1. **分轮确认**：每一步操作前展示变更摘要，用户明确回复"确认/继续"后才执行，不在同一轮次同时展示确认问句和执行操作
2. **先读再写**：所有 `modify_*` 操作前必须先调 `describe_*` 读取当前值，避免覆盖已有配置
3. **二次确认**：`create_app`、`release_app`、`import_workflow` 等高风险操作必须包含风险提示+改动摘要+确认问句三要素
4. **写后验证**：`modify_agent_skill_list` / `modify_agent_tool_list` 执行后，再调 `describe_agent_config` 验证变更是否生效
5. **批量操作间隔**：连续操作间隔 3 秒以上，避免 403 限流
6. **长文本传参**：提示词等长文本用 `-f path.json` 传参，避免 `ARG_MAX` 限制
7. **skill_id 真实性**：挂载 Skills 时必须用真实 ID，不能用显示名称代替
