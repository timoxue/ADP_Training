# 案例对照：投研助手 vs 财富顾问助手 vs 投研小团队

本文件提供三个完整的 ADP 应用开发案例，供六步推理链每一步做参照。

---

## §1 第一步：业务产出物

### 投研助手

```
应用名称：投研助手
目标用户：基金经理/研究员（培训学员）
核心场景：分析上市公司财务指标、关联交易、研报观点，学习 AI 能力边界
交付物形态：
  - 关键指标对照表（三阶递进：知识库原文→API交叉校验→校验结论）
  - 关联交易提取清单（三阶递进：入门→进阶→专业/合规）
  - 实时资讯（新闻+微信观点）
成功标准：学员能直观看到知识库与 API 数据差异，理解 AI 交叉校验的价值
```

### 财富顾问助手

```
应用名称：财富顾问助手
目标用户：银行理财客户
核心场景：查产品、比费率、看适配性、获取投资建议
交付物形态：
  - 【直接回答】：产品推荐/问题解答
  - 【产品依据】：匹配依据+费率对比
  - 【风险提示】：合规风险提示
成功标准：推荐产品与客户风险等级匹配，四条合规红线不踩
```

### 投研小团队

```
应用名称：投研小团队
目标用户：基金经理/研究员
核心场景：给一个股票代码，多线程并行完成——基本面分析、行业对比、新闻舆情、风险预警，最后汇总成投研快报
交付物形态：多Agent并行分析 → 汇总监整报告
  - 【财务分析】核心指标+同比变化+财务风险评估
  - 【行业对比】对标公司清单+关键指标对比+行业位置评估
  - 【舆情动态】最新新闻TOP5+深度文章TOP3+舆情风向
  - 【风险预警】风险信号清单(🔴🟡🟢)+关联交易+异常事件
  - 【综合结论】调度Agent整合判断
成功标准：比单人逐一查快3-5倍，输出结构完整可交付
```

## §4 第四步：设计推理

### 投研助手关键决策

**决策0：应用模式选择**
- 选：**Claw 模式**——需要灵活调用 a-stock-data/news/wechat-article-search 等多个 Skills，Skills 即装即用最方便；需要独立工作空间做数据分析
- 不选：标准模式——投研分析需要灵活调用多种数据源，标准模式下这些数据源需要通过连接器/工作流实现，配置更复杂；且投研场景不需要双模型协同的"稳定精确"优势（学员恰恰要看到 AI 的能力边界）
- ⚠️ **Claw 模式的代价**：没有内容安全/保守回答/隐私脱敏/长期记忆/上下文轮数等平台级设置项。投研场景是培训用途，无合规硬拦截需求，所以可以接受。但如果未来需要合规硬拦截，只能新建标准模式应用。
- 边界：Claw 模式不支持工作流——未来如果想做"定期投研巡检"这类固定流程，无法在同应用内实现
- 兜底：需要固定流程时新建一个标准模式应用，或用提示词中的"追问"机制模拟递进流程

**决策1：知识库 vs 实时 API**
- 选：两者结合——知识库存年报原文（结构化但滞后），API 补实时数据（最新但需校验）
- 不选：纯知识库（数据滞后）或纯 API（无法提取年报原文的表述细节）
- 边界：知识库文档覆盖范围有限（只有入库的公司）
- 兜底：巨潮在线检索补未入库公司公告（searchkey="年度报告"/"关联交易")

**决策2：数据源优先级**
- 选：新浪 HTTP 接口为主（稳定无反爬）
- 不选：mootdx TCP 7709（沙箱环境不通）、东财（JS反爬+限流）
- 边界：新浪接口字段可能少于 mootdx
- 兜底：字段缺失时标注"该数据源暂不可获取"

### 财富顾问助手关键决策

**决策0：应用模式选择**
- 选：**标准模式**——合规场景必须有平台级硬拦截（内容安全+保守回答+隐私脱敏），财富顾问四条红线不能只靠提示词软约束
- 不选：Claw 模式——❌ **Claw 模式没有内容安全/保守回答/隐私脱敏设置项**，合规红线可被 prompt injection 绕过，风险不可接受
- 边界：标准模式知识库原生集成，比 Claw 的 Skill 调用更稳定精确——对合规场景反而是优势
- 兜底：标准模式可切换到单工作流/Multi-Agent，未来扩展灵活

**决策1：客户数据接入方式**
- 选：API 传参（少量固定字段：姓名/风险等级/CustomerID）+ 长期记忆（跨会话画像）
- 不选：每次对话手动输入客户信息（体验差）
- 边界：API 只能传几个固定字段，大量结构化数据（持仓/交易记录）不适合
- 兜底：后续可考虑知识库连数据库（NL2SQL）扩展

**决策2：风控实现方式**
- 选：三层防线——平台强制（内容安全+隐私脱敏）→ 提示词软约束（四条红线+保守回答）→ 人工复核
- 不选：只靠提示词（可被绕过）或只靠平台（不够灵活）
- 边界：平台拦截可能误伤正常咨询；提示词约束可能被 prompt injection 绕过
- 兜底：人工复核是最终防线

**决策3：长期记忆方案**
- 选：`{{SYS.Memory}}` 注入客户画像 + `{{API.CustomerID}}` 隔离记忆
- 不选：不用记忆（每次重新介绍自己，体验差）
- 边界：记忆可能随时间过时
- 兜底：提示词要求"记忆可能过时，以本次对话为准"

### 投研小团队关键决策

**决策0：应用模式选择**
- 选：**Multi-Agent 模式**——多角色并行是核心诉求，财务分析师/行业分析师/舆情专员/风控专员同时干活，单Agent只能串行
- 不选：Claw模式——单Agent无法真正并行，体现不出"团队"优势；标准模式——无多角色协作机制
- ⚠️ **Multi-Agent 的代价**：调度Agent可能误判问题类型分错子Agent；子Agent输出格式不统一会导致汇总丢信息
- 边界：子Agent数量不超过5个，超过调度复杂度爆炸
- 兜底：调度Agent提示词中明确定义转交条件和关键词触发规则，子Agent输出格式标准化

**决策1：Multi-Agent vs 单Agent角色分级**
- 选：Multi-Agent——真正并行处理，4个子Agent同时干活，速度3-5倍
- 不选：Claw+提示词角色分级——只能串行模拟，无法真正并行，且角色切换不灵活
- 边界：Multi-Agent子Agent间无法直接共享中间结果，只能通过调度Agent传递
- 兜底：各子Agent输出标准化格式（Markdown标题+编号列表），调度Agent解析汇总

**决策2：子Agent数量与分工**
- 选：4个子Agent+1个调度Agent = 5个
- 不选：更多（如拆出"估值分析师"）→ 管理复杂度太高
- 边界：4个已经是Multi-Agent的合理上限
- 兜底：某个子Agent处理不了时转回调度Agent重新分配

**决策3：数据源分配**
- 选：按子Agent分工分配数据源（财务→a-stock-data+dcf-model；行业→a-stock-data+data-analysis；舆情→news+wechat+fact-check；风控→a-stock-data+kb-search）
- 不选：所有Agent共享所有数据源 → 权限混乱
- 边界：a-stock-data的skill_id只能挂载到整个应用，子Agent都能调用
- 兜底：提示词约束各子Agent只用自己领域的数据源
