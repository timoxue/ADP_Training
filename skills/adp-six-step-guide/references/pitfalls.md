# 常见坑与避坑指南

本文件是第六步「验收」的参考，也建议在操作前预读，提前规避常见问题。

---

## ⚠️ 最大坑：Claw 模式没有合规硬拦截设置项

> **这是模式选择阶段最容易踩的坑，也是影响最深远的坑。**

### 坑 0：选了 Claw 模式后发现需要合规硬拦截——只能新建应用从头来
- **现象**：搭建了 Claw 模式应用后，想开启"内容安全""保守回答""隐私脱敏"等设置，发现应用设置里根本没有这些选项
- **原因**：Claw 模式的应用设置仅有 4 项：欢迎语、对话体验、变量、高级设置(同义词)。**内容安全、保守回答、隐私脱敏、长期记忆、上下文轮数、模型设置（双模型体系）——这些全部不存在。**
- **官方文档佐证**：
  1. 应用设置概述文档明确标注"本文以标准模式应用为例"——所有设置项描述都是标准模式的
  2. 四种模式文档指出"Claw 模式的工作空间、Skills、连接器等配置与其他三种模式差异较大"
  3. Claw 模式不可切换——一旦创建就无法改到标准模式
- **影响**：
  - 合规场景（金融/医疗/法律）需要平台级硬拦截（内容安全+保守回答+隐私脱敏），Claw 模式只能靠提示词软约束，可被 prompt injection 绕过
  - 长期记忆场景（跨会话记住用户偏好）无法实现
  - 长对话场景无法调整上下文轮数
- **规避**：在第三步模式选择时，必须问清楚——"你的应用需要合规硬拦截吗？"需要 → ❌ 不能选 Claw，选标准/单工作流/Multi-Agent
- **案例**：财富顾问助手（合规场景）必须用标准模式——四条红线（不承诺收益/不说保本/不代客操作/超范围不作答）需要平台级内容安全+保守回答双重拦截，Claw 模式做不到

## 知识库类

### 坑 1：PDF 表格丢结构
- **现象**：上传的 PDF 年报中的表格，检索时字段错位、丢失、乱码
- **原因**：PDF 解析引擎对复杂表格支持有限，尤其是合并单元格
- **规避**：验收时用实际查询检验；关键表格考虑提供 Excel/CSV 版本补充
- **案例**：投研助手的关联交易公告表格解析时丢失列结构，需提示学员注意

### 坑 2：文档解析状态未完成就测试
- **现象**：上传文档后立即测试，知识库检索不到内容
- **原因**：文档还在解析中（解析中→审核中→学习中→导入完成）
- **规避**：用 `list_doc.py --bot-biz-id {app_id}` 确认所有文档状态为"导入完成"

### 坑 3：FAQ 导入格式不对
- **现象**：FAQ 文件上传失败或内容乱码
- **原因**：`--opt 1` 问答对导入不支持 MD 格式
- **规避**：转 CSV 或 Excel 格式上传

### 坑 4：检索配置不当
- **现象**：检索结果太多噪音 / 太少漏检
- **原因**：Top-K 或相似度阈值配置不当
- **规避**：投研场景建议 Top-K 5-10，相似度 0.5-0.7，开引用溯源，用混合检索

## API/数据源类

### 坑 5：mootdx TCP 7709 不通
- **现象**：A 股数据调用失败
- **原因**：mootdx 走 TCP 协议，部分沙箱/容器环境封锁 TCP 7709 端口
- **规避**：以新浪 HTTP 接口为主（稳定无反爬），mootdx 作为备选

### 坑 6：东财接口被限流/封 IP
- **现象**：东财数据接口返回空或 403
- **原因**：东财有反爬机制，频繁调用触发封 IP
- **规避**：优先用通达信/腾讯接口（不封 IP）；东财接口内置限流防封，间隔调用

### 坑 7：巨潮公告搜索找不到结果
- **现象**：`cninfo_announcements(code, searchkey="关联交易")` 返回空
- **原因**：searchkey 关键词不匹配
- **规避**：实测有效关键词：`"关联交易"`、`"年度报告"`、`"问询函"`

## 应用管理类

### 坑 8：skill_id 用了显示名称
- **现象**：`modify_agent_skill_list` 调用后技能没挂上
- **原因**：传了 `"pdf"` 而不是真实 ID `"xxx-xxx-xxx"`
- **规避**：先调 `describe_skill_list` 拿真实 ID，`add_skill_list` 格式是 `["id1", "id2"]`（字符串数组）

### 坑 9：提示词被整段覆盖
- **现象**：修改提示词后原有配置丢失
- **原因**：平台是整段替换语义，没先读当前值就直接写
- **规避**：**先读再写**——先 `describe_agent_config` 拿当前提示词，展示 diff 后再修改

### 坑 10：开场问题超过 3 条
- **现象**：`modify_app_greeting` 调用返回参数错误
- **原因**：`opening_question_list` 最多 3 条
- **规避**：控制在 3 条以内

### 坑 11：403 限流
- **现象**：批量操作时部分请求返回 403
- **原因**：短时间内大量 API 调用触发限流
- **规避**：操作间隔 3 秒以上；`describe_skill_list`/`describe_plugin_list`/`create_app` 报 403 时检查是否带了 `space_id`，带上后重试 1 次

### 坑 12：长期记忆开关找不到 / Claw 模式根本没有长期记忆
- **现象**：用户在 Agent 编辑页找不到长期记忆开关
- **原因（标准模式）**：长期记忆不在 Agent 编辑页，在「应用设置」独立菜单
- **原因（Claw 模式）**：Claw 模式的应用设置里**根本没有长期记忆这个选项**
- **规避**：
  - 标准模式：正确路径——Agent 编辑页 → 右上角「应用设置」→ 左侧菜单「长期记忆」
  - Claw 模式：**不存在此功能**，如需长期记忆必须选标准模式

### 坑 13：共享知识库 API 创建
- **现象**：想用 API 创建共享知识库
- **原因**：共享知识库只能在 ADP 控制台手动创建
- **规避**：引导用户在控制台创建（ADP平台→左侧「知识库」→「新建知识库」）

## 提示词类

### 坑 14：提示词太长导致模型遗忘
- **现象**：长提示词中后面的约束没有生效
- **原因**：提示词过长，模型注意力衰减
- **规避**：提示词控制在 8000 字符以内；重要约束放在开头和结尾（首因效应+近因效应）

### 坑 15：API 变量未注入
- **现象**：`{{API.CustomerName}}` 等变量显示为字面文本
- **原因**：API 调用时未传入 `custom_vars`
- **规避**：确保 API 调用参数中包含 `custom_vars: {"CustomerName": "...", "RiskLevel": "...", "CustomerID": "..."}`

## Multi-Agent 类

### 坑 16：转交描述重叠导致调度混乱
- **现象**：用户问一个问题，调度Agent在两个子Agent之间反复转交，或同时转给多个不该接收的子Agent
- **原因**：两个子Agent的转交描述都包含相同的关键词（如都写了"分析"）
- **规避**：转交描述必须互斥——每个子Agent使用领域专属关键词（如"财务指标"vs"行业对比"vs"新闻舆情"vs"风险预警"）；转交描述不宜过长，调度Agent用关键词匹配，写太多反而稀释匹配度

### 坑 17：子Agent输出格式不统一导致汇总丢信息
- **现象**：调度Agent汇总时丢失子Agent的关键数据，或汇总格式混乱
- **原因**：各子Agent输出格式不一致，调度Agent无法正确解析
- **规避**：每个子Agent必须用标准化输出模板（Markdown标题+编号列表）；调度Agent提示词中定义汇总模板，直接引用子Agent输出

### 坑 18：Multi-Agent 知识库问答未启用
- **现象**：子Agent调用知识库检索时返回空或报错
- **原因**：Multi-Agent 模式下知识库问答不是默认启用，需手动启用"知识库问答"工具
- **规避**：在 Agent 编辑页 → 工具 → 添加"知识库问答"工具

### 坑 19：子Agent数量过多导致调度复杂度爆炸
- **现象**：5个以上子Agent时调度Agent频繁误判，响应时间变长
- **原因**：每增加一个子Agent，调度复杂度指数增长
- **规避**：子Agent数量建议不超过5个；如需更多角色，考虑拆成两个Multi-Agent应用

## 发布类

### 坑 20：发布后修改不生效
- **现象**：修改配置后用户端没变化
- **原因**：修改后未重新发布
- **规避**：每次修改后都需要调 `release_app` 重新发布

### 坑 21：工作流导入失败
- **现象**：`import_workflow` 失败
- **原因**：工作流包格式不对或版本不兼容
- **规避**：使用 `adp-workflow-package-generator` 的脚本生成，不要手写 JSON；生成后先运行 `verify_delivery.py` 校验

---

## 安全审查类

### 坑 22：安装第三方技能前未做安全检查
- **现象**：安装了来源不明的技能，可能包含恶意代码
- **原因**：跳过了安全审查步骤
- **规避**：安装任何第三方技能前，先用 `skills-security-check` 审计；P0 级别禁止安装，P1 需用户确认，P2 可安全安装
