# 常见设计决策对比与选择指南

本文件是第四步「设计推理」的详细参考，帮助在多种方案间做出有依据的选择。

---

## 决策 1：知识库 vs 实时 API

| 维度 | 知识库 | 实时 API |
|---|---|---|
| 数据时效 | 滞后（取决于上传频率） | 实时 |
| 数据精度 | 原文级（保留表述细节） | 字段级（可能丢失上下文） |
| 可靠性 | 稳定（本地检索） | 依赖网络和接口可用性 |
| 成本 | 上传和维护文档的人力 | API 调用可能有限流/费用 |
| 适合内容 | 年报、招股书、产品说明书、规则文档 | 行情数据、新闻、公告 |

**选择建议**：
- 静态文档（年报、说明书、FAQ）→ 知识库
- 实时数据（股价、新闻）→ API
- 需要交叉校验（知识库 vs API 对比找差异）→ 两者结合

**投研助手案例**：两者结合是亮点——学员能直观看到知识库原文与 API 数据的差异，理解精度和口径问题。

## 决策 2：标准模式 vs 工作流模式

| 维度 | 标准模式 | 工作流模式 |
|---|---|---|
| 交互方式 | 自由对话 | 固定流程步骤 |
| 灵活性 | 高（用户自由提问） | 低（按流程走） |
| 可控性 | 低（靠提示词约束） | 高（流程节点可控） |
| 适合场景 | 咨询/分析/问答 | 审批/采集/多步操作 |
| 技能需求 | adp-prompt-builder | adp-workflow-package-generator |

**选择建议**：
- 用户需求是"问什么答什么"→ 标准模式
- 用户需求是"按步骤走完一个流程"→ 工作流模式
- 两者结合：主体用标准模式，特定子任务用工作流

## 决策 3：提示词约束 vs 平台安全

| 维度 | 提示词约束（软） | 平台设置（硬） |
|---|---|---|
| 强制力 | 可被 prompt injection 绕过 | 平台层强制，不可绕过 |
| 灵活性 | 高（随意调整措辞） | 低（开关式，粒度粗） |
| 可见性 | 用户可看到约束逻辑 | 用户不可见内部机制 |
| 适合内容 | 业务规则、输出格式、角色设定 | 敏感内容拦截、隐私脱敏 |

**选择建议**：
- 内容类约束（输出格式、角色设定、业务规则）→ 提示词
- 风控类约束（敏感内容、隐私保护、合规红线）→ 平台设置
- **最佳实践：软硬结合，提示词做第一道，平台做最后一道**

**财富顾问助手案例**：三层防线——平台强制（内容安全+隐私脱敏）→ 提示词软约束（四条红线）→ 人工复核。

## 决策 4：API 传参 vs 知识库连数据库

| 维度 | API 传参 | 知识库连数据库 |
|---|---|---|
| 数据量 | 少量固定字段 | 大量结构化数据 |
| 查询方式 | 固定变量注入 | NL2SQL 自然语言查库 |
| 实时性 | 每次调用注入最新 | 取决于数据库同步 |
| 适合内容 | 用户身份、偏好、状态 | 产品目录、交易记录、持仓明细 |
| 技术门槛 | 低（API 参数） | 中（需建表+配置数据库连接） |

**选择建议**：
- 少量固定字段（姓名、风险等级、ID）→ API 传参
- 大量结构化查询（产品费率、持仓明细、交易记录）→ 知识库连数据库
- 两者可组合使用

**财富顾问助手案例**：API 传参用于 `{{API.CustomerName}}`/`{{API.RiskLevel}}`，未来产品数据库用知识库连数据库。

## 决策 5：单一 Agent vs Multi-Agent

| 维度 | 单一 Agent | Multi-Agent |
|---|---|---|
| 复杂度 | 低 | 高 |
| 职责清晰度 | 一个角色做所有事 | 每个角色专注一件事 |
| 适合场景 | 职责单一的助手 | 多角色协作（如：研究员+销售+基金经理） |
| 提示词长度 | 一段完整提示词 | 每个子 Agent 各一段 |
| 调度方式 | 无需调度 | 需要调度 Agent 做路由 |

**选择建议**：
- 职责单一、角色清晰 → 单一 Agent
- 多角色协作、需分工 → Multi-Agent
- 可通过提示词中的角色分级实现"轻量多角色"（如投研助手的三级角色：研究员/销售/基金经理）

**投研助手案例**：单一 Agent + 提示词角色分级，通过对话识别当前角色，无需 Multi-Agent 架构。

**投研小团队案例**：Multi-Agent，4个子Agent（财务分析师/行业分析师/舆情专员/风控专员）+ 1个调度Agent（投研队长），真正并行处理，速度3-5倍。选择Multi-Agent而非Claw+角色分级的原因：投研小团队需要真正并行而非串行模拟。

## 决策 6：知识库检索配置

| 参数 | 推荐值 | 说明 |
|---|---|---|
| 检索模式 | 混合检索（向量+关键词） | 兼顾语义理解和精确匹配 |
| Top-K | 5-10 | 跨文档交叉引用需要较多结果 |
| 相似度阈值 | 0.5-0.7 | 太低噪音多，太高漏检多 |
| 引用溯源 | 必须开启 | 让用户知道信息来自哪个文档 |

## 决策 7：长期记忆方案

| 方案 | 适合场景 | 配置方式 |
|---|---|---|
| 不开记忆 | 一次性查询场景 | 无需配置 |
| `{{SYS.Memory}}` | 需要跨会话记住用户信息 | 应用设置→长期记忆→开启 |
| API 传参+记忆 | 需要按客户 ID 隔离记忆 | API 传 `CustomerID` + 长期记忆 |

**关键提醒**：长期记忆开关不在 Agent 编辑页，在「应用设置」独立菜单项（右上角→应用设置→左侧菜单→长期记忆）。

## 决策 8：合规治理强度

> **合规是设计前提，不是事后补丁。** 详见 `references/compliance-governance.md`

| 场景类型 | 合规强度 | 三条红线适用 | 需要硬拦截 | 推荐模式 |
|---|---|---|---|---|
| 内部培训/工具 | 🟢 低 | 不适用 | ❌ 不需要 | 任意模式 |
| 内部投研/风控 | 🟡 中 | 部分适用（不承诺收益） | ⚠️ 按需 | Claw 可接受 |
| 面向客户金融 | 🔴 高 | 全部适用 | ✅ 必须 | 标准模式（Claw ❌） |

**三条不可逾越的表述红线**：
- ❌ 不承诺收益（"保证年化8%"→拦截）
- ❌ 不代客操作（"帮你买入"→拦截）
- ❌ 不构成投资建议（"强烈推荐"→拦截）

**固化手段三层兜底**：

| 层级 | 机制 | 作用 |
|---|---|---|
| 提示词约束（软） | 角色指令写红线 | 第一道防线，可被绕过 |
| 内容安全黑词拦截（硬） | 命中关键词自动替换 | 第二道防线，不可绕过 |
| 回复模板免责声明（兜底） | 每次输出自动附免责 | 第三道防线，兜底 |

## 决策 9：检出 vs 拦截 vs 脱敏

> **检出 ≠ 拦截 ≠ 脱敏：三件事职责不同，要搭配使用。** 详见 `references/compliance-governance.md` §三

| 动作 | 定义 | 解决什么问题 | ADP 实现方式 |
|---|---|---|---|
| **检出** | 识别出内容中存在风险信息 | "知道有" | 提示词要求标注风险 → 写入《风险预检清单》供人工复核 |
| **拦截** | 阻止风险内容输出给用户 | "不让出去" | 内容安全 → 命中关键词自动替换为处置文案 |
| **脱敏** | 对敏感信息打码处理 | "打码" | 隐私脱敏 → 身份证/银行账户等自动打码 |

**常见错误**：
- 只做检出不做拦截 → 风险内容仍然输出给用户
- 只做拦截不做检出 → 不知道有什么风险，无法改进
- 用检出代替脱敏 → 风险信息仍以明文输出
- 只靠提示词做拦截 → prompt injection 可绕过

**正确搭配**：检出（标记风险）+ 拦截（阻断红线）+ 脱敏（打码敏感信息），三层独立运作，互相补充。
