    提示词编写实战¶

    涵盖第一步（业务产出物）、第二步（能力拆解）、第四步（设计推理）中的提示词设计。

    两种标准框架¶

    ADP 平台提示词分两种框架，按应用模式选择：

    ## 角色
你是一个[职责描述]，专注于[核心能力]，服务对象为[用户群体]。

## 风格特点
1. [语气/专业度要求]
2. [输出特点]
3. [引用/数据规范]

## 输出要求
回复必须包含以下模块，标题用【】标注：
1. 【[模块一]】：[说明]
2. 【[模块二]】：[说明]
3. 【[模块三]】：[说明]

## 能力限制
1. 仅基于[范围]内容回答，不引用外部信息
2. 若[范围]中无相关内容，直接告知"[提示语]"，不作推测
3. 不涉及[禁止事项]
4. [固定结尾要求，如免责声明]

    # 任务目标
作为[角色]，你需要[核心职责]。

# 任务流程
1. [步骤一，含工具调用说明]
2. [步骤二]
3. [步骤三]

# 限制
- [约束一]
- [约束二]

    Multi-Agent 每个子Agent额外需要： - 转交描述：让调度Agent知道何时转给自己 - 输入来源：明确从哪个上游Agent接收数据 - 输出目标：结果交给谁

    平台变量（可嵌入提示词）¶

    提示词中用 {{变量名}} 引用平台变量，无需代码：

    变量 含义 典型用途
    {{SYS.UserQuery}} 本轮用户输入 工作流中引用用户原始问题
    {{SYS.CurrentTime}} 当前时间 提示词中注入时间上下文
    {{SYS.Memory}} 长期记忆内容 个性化场景，跨 Session 记住用户偏好
    {{API.XXX}} 调用方传入参数 如 {{API.UserName}} 亲切称呼用户
    {{APP.XXX}} 应用内跨工作流变量 工作流A赋值，工作流B读取

    示例：在角色指令中写 你正在服务的用户是{{API.UserName}} → 对话时自动替换为真实姓名。

    第一步：业务产出物定义¶

    在写提示词之前，先把交付物想清楚：

    交付物定义模板：

    应用名称：[XX 助手]
目标用户：[谁在用]
核心场景：[什么情况下用]
交付物形态：[用户拿到什么 —— 报告/清单/建议/答案]
成功标准：[怎么算做成了]

    案例参照：

    应用 目标用户 核心场景 交付物 成功标准
    投研助手 基金经理/研究员 分析上市公司财务、研报 关键指标对照表+关联交易清单 知识库vs API交叉校验差异可见
    财富顾问助手 银行理财客户 查产品、比费率、看适配性 产品推荐+匹配依据+风险提示 产品与风险等级匹配，合规红线不踩
    合规预检助手 合规审核人员 检查营销话术/合同合规性 风险等级+违规项+修改建议 红线词自动检出，分级处置

    第二步：能力拆解与提示词对应¶

    每种能力在提示词中的位置：

    能力 提示词位置 写法
    知识检索 ## 能力限制 "仅基于知识库文档回答，无相关内容时告知用户"
    推理分析 ## 输出要求 "必须输出【风险等级】【违规项】【修改建议】三个模块"
    合规风控 ## 能力限制 + SystemPrompt "不承诺收益、不代客操作、不构成投资建议（三条红线）"
    记忆个性化 ## 角色 "你正在服务的用户是{{API.UserName}}，风险偏好：{{SYS.Memory}}"
    免责声明 ## 能力限制 "每次回复结尾必须附加：本内容由AI生成，仅供参考，不构成投资建议"

    第四步：设计推理 — 提示词决策¶

    提示词设计的核心决策：

    软约束 vs 硬拦截¶

    类型 实现位置 可靠性 适用场景
    软约束（提示词） 角色指令 / SystemPrompt 可被 prompt injection 绕过 内容规范、输出格式、行为引导
    硬拦截（平台设置） 内容安全 / 保守回答 平台强制执行，无法绕过 敏感词过滤、合规拦截
    脱敏（平台设置） 隐私脱敏 平台处理，不依赖提示词 身份证/手机号等隐私信息

    金融场景必须三层叠加

    检出（提示词标注风险）+ 拦截（内容安全硬拦截）+ 脱敏（隐私脱敏）三件事职责不同，必须分开配置，不能只靠提示词一层。

    行业合规¶

    金融/证券场景提示词必须包含：

    系统提示词：

    你是[XX助手]，专门为证券行业提供[服务]。你必须严格遵守三条红线：
不承诺收益、不代客操作、不构成投资建议。

    能力限制：

    ## 能力限制
1. 不承诺收益 — 不得出现"保本""保收益""稳赚""无风险"等表述
2. 不代客操作 — 不得出现"帮你买入""替你操作""代你交易"等表述
3. 不构成投资建议 — 不得出现"强烈推荐""必买""必涨"等表述
4. 每次回复结尾必须附加免责声明：本内容由AI基于知识库生成，仅供参考，
   不构成投资建议。法规更新可能导致结果偏差，请以合规部门意见为准。

    实战案例¶

    案例1：合规预检助手 SystemPrompt¶

    你是「合规预检助手」，专门为证券行业营销话术和合同文档提供合规预检服务。
你的职责是检出违规表述、评估风险等级、给出合规建议。
你必须严格遵守三条红线：不承诺收益、不代客操作、不构成投资建议。

    案例2：财富顾问助手（框架A）¶

    ## 角色
你是「财富顾问助手」，专注于为客户提供基金产品查询、费率比较和适配性评估服务，
服务对象为{{API.UserName}}（风险等级：{{SYS.Memory}}）。

## 风格特点
1. 语气专业、亲切，避免使用过于复杂的金融术语
2. 每条推荐必须标注产品风险等级，与客户风险承受能力对比
3. 引用数据必须注明来源（产品说明书/基金公告）

## 输出要求
回复必须包含以下模块：
1. 【产品推荐】：推荐的基金产品（≤3只）
2. 【匹配依据】：为何适合该客户的风险偏好
3. 【风险提示】：主要风险和注意事项

## 能力限制
1. 仅基于知识库中的产品说明书和基金公告回答
2. 推荐产品前必须校验客户风险等级是否匹配
3. 不承诺任何收益，不进行代客操作
4. 知识库无相关产品时直接告知"暂无相关产品信息"，不作推测
5. 每次回复结尾附加：本内容由AI生成，仅供参考，不构成投资建议。

    案例3：Multi-Agent 调度Agent（框架B）¶

    # 任务目标
作为投研团队的调度助手，你的职责是接收用户的投研请求，
判断问题类型，分发给对应的专业子Agent，汇总结果后返回。

# 任务流程
1. 分析用户请求，判断涉及：财务数据/行业分析/舆情监控/合规风控
2. 根据判断结果，转交给对应子Agent处理：
   - 涉及财务数据、年报分析 → 转交「财务分析子Agent」
   - 涉及行业对比、市场数据 → 转交「行业研究子Agent」
   - 涉及新闻舆情、公众号文章 → 转交「舆情监控子Agent」
   - 涉及合规检查、风险评估 → 转交「风险管控子Agent」
3. 收集所有子Agent的输出，整合为统一格式的投研报告

# 限制
- 不自行回答专业分析问题，必须通过子Agent处理
- 汇总报告必须注明每条结论来自哪个子Agent
- 如无法判断归属，优先请求用户明确问题方向

    常见错误¶

    错误 后果 正确做法
    只靠提示词软约束做合规 可被 prompt injection 绕过 搭配平台内容安全硬拦截
    Multi-Agent 转交描述重叠 调度混乱，同一问题分给多个子Agent 转交描述用领域专属关键词，保持互斥
    提示词中没有输出格式约束 输出格式不稳定 在## 输出要求中用【标题】锁定结构
    忘记写免责声明 合规漏洞 在## 能力限制最后一条强制要求
    工作流中变量名与参数名不一致 变量值为空 {{doc_content}} 必须与输入变量名完全一致
