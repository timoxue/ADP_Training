    六步分析法详解¶

    六步推理链是一条线性递进的思考链——每一步的输出是下一步的输入。跳步 = 返工。

    交互原则¶

    1. 每一步先确认再前进：完成一步后展示结论，确认无误后再进入下一步
    2. 允许回溯：随时可以回到第X步重新推导
    3. 不跳步不省略：即使"直接帮我搭"，也至少要快速确认前四步结论
    4. 结合案例讲解：每一步用「投研助手」和「财富顾问助手」对照说明

    第一步：业务产出物¶

    核心问题：我最终要交付什么？

    引导用户回答：

    1. 你的用户是谁？（终端用户画像）
    2. 用户在什么场景下会用这个应用？
    3. 用户完成一次对话后，拿到什么？
    4. 怎么算"好用"？（验收标准雏形）

    交付物定义模板：

    应用名称：[XX 助手]
目标用户：[谁在用]
核心场景：[什么情况下用]
交付物形态：[用户拿到什么 —— 报告/清单/建议/答案/...]
成功标准：[怎么算做成了]

    案例对照：

    维度 投研助手 财富顾问助手
    目标用户 基金经理/研究员 银行理财客户
    核心场景 分析上市公司财务、关联交易、研报 查产品、比费率、看适配性
    交付物 关键指标对照表+关联交易清单+实时资讯 产品推荐+匹配依据+风险提示
    成功标准 知识库vs API交叉校验差异可见 推荐产品与风险等级匹配，合规红线不踩

    第二步：能力拆解¶

    核心问题：要产出它，需要哪几种能力？

    常见能力维度：

    能力类别 说明 典型表现
    知识检索 从自有文档中找信息 "从年报中提取财务数据"
    数据查询 从外部数据源获取实时数据 "查询最新股价、财务三表"
    推理分析 基于数据做交叉校验、对比 "知识库数据与 API 数据对比"
    内容生成 按模板/框架组织输出 "生成关键指标对照表"
    合规风控 检查边界、兜底 "不承诺收益、风险提示"
    合规治理 检出/拦截/脱敏三层 "内容安全硬拦截+隐私脱敏+免责声明"
    记忆与个性化 跨会话记住用户信息 "记住客户风险偏好"
    外部系统集成 对接第三方平台 "连接邮箱、文档平台"

    拆解模板：

    交付物：[从第一步]
能力清单：
  1. [能力1]：[具体描述]
  2. [能力2]：[具体描述]
能力优先级：P0(必须有) / P1(锦上添花) / P2(未来扩展)

    案例对照：

    能力 投研助手 财富顾问助手
    知识检索 P0：从年报/研报提取数据 P0：从产品说明书提取信息
    数据查询 P0：A股行情/财务三表 P1：基金净值
    推理分析 P0：知识库vs API交叉校验 P0：风险等级匹配校验
    内容生成 P0：按三阶递进生成对照表 P1：生成产品推荐报告
    合规风控 P1：标注监管依据 P0：四条红线+保守回答
    记忆个性化 P2 P0：跨会话记忆客户画像

    第三步：功能映射 + 模式选择¶

    核心问题：每种能力对应平台哪个功能？选择哪种应用模式？

    详见 功能映射与模式选择。

    映射方法：

    第一轮：经验初筛 → 候选列表
第二轮：平台搜索验证（describe_skill_list / describe_plugin_list）→ 补充遗漏技能
第三轮：匹配度排序（★★★完全匹配 / ★★☆高度匹配 / ★☆☆部分匹配）→ 最终映射表

    不要只靠经验映射

    ADP 平台有 100+ Skills，务必执行第二轮平台搜索，避免遗漏高匹配度技能。

    第四步：设计推理¶

    核心问题：为什么选它？边界与兜底在哪？

    对每个映射关系做四维论证：

    推理模板：

    能力：[XXX]
选它：[为什么选这个功能/技能]
不选：[考虑过但放弃的替代方案，为什么放弃]
边界：[这个方案做到什么程度就到极限了]
兜底：[超出边界时怎么处理]

    常见设计决策：

    决策点 选择逻辑
    知识库 vs 实时 API 静态文档用知识库，实时数据用 API，交叉校验用两者结合
    标准模式 vs 工作流模式 简单对话用标准，多步骤流程用工作流
    提示词约束 vs 平台安全 内容类约束放提示词（软），风控类约束放平台设置（硬）
    单一 Agent vs Multi-Agent 职责清晰用单一，多角色/多流程用 Multi-Agent
    合规治理强度 金融/医疗必须硬拦截（不能选Claw模式）；内部工具合规要求低
    检出/拦截/脱敏 三件事职责不同，必须搭配，不能只靠一层

    第四步结束时必须输出搭建模式推荐结论：

    📋 搭建模式推荐结论

推荐模式：[Claw/标准/单工作流/Multi-Agent]
推理依据：
  1. [来自设计推理的依据1]
  2. [来自设计推理的依据2]
替代方案：[退而求其次选什么，代价是什么]
⚠️ 关键风险：[需要特别注意什么]
🔒 合规风险：[合规治理强度评估——高/中/低]

    为什么推荐结论在第四步输出

    第三步做能力映射和模式分析，输出"候选模式"。第四步把每个映射的选它/不选/边界/兜底想清楚后，才能给出有依据的最终推荐。结论必须有推理支撑。

    第五步：分步操作¶

    核心问题：怎么搭？

    详见 应用创建与配置。

    流程：先展示完整手工操作清单（5A）→ 用户确认理解 → 再询问搭建方式（手工/半自动/全自动）

    正确流程

    不能在展示手工分解的过程中就问"要不要我帮你搭"。必须先展示完整清单，用户确认理解后，再提供搭建方式选择。

    第六步：验收¶

    核心问题：做完怎么验对错？

    验收用例模板：

    用例编号：TC-001
用例名称：[一句话描述]
输入：[用户说什么]
预期输出：[应用应该返回什么]
验收标准：[通过/失败的具体判断条件]

    验收维度：

    维度 检查项 适用场景
    功能正确性 核心能力是否可用 全部
    边界兜底 超范围问题是否有合理兜底 全部
    合规安全 风控约束是否生效 全部
    合规红线 三条红线（不承诺收益/不代客操作/不构成投资建议） 金融场景必测
    检出/拦截/脱敏 三层是否独立运作 标准/单工作流/Multi-Agent
    免责声明 每次输出是否自动附带 金融场景必测
    调度准确性 调度Agent是否正确判断并转交 Multi-Agent
    工作流执行 节点是否按预期顺序执行 单工作流

    高频踩坑：

    # 坑 后果 避法
    1 知识库文档解析失败 PDF表格丢结构 验收时用实际查询检验
    2 skill_id 传错显示名称 操作无效 用真实 ID
    3 提示词整段替换 覆盖已有配置 先读后写
    4 开场问题超过3条 平台拒绝 最多3条
    5 403 限流 操作失败 间隔3秒以上
    6 只靠提示词做合规拦截 可被绕过 搭配内容安全硬拦截
    7 Multi-Agent 转交描述重叠 调度混乱 转交描述必须互斥
    8 检出/拦截/脱敏混为一谈 合规漏洞 三件事分开配置

    整体交付物¶

    完成六步后输出《应用开发设计文档》：

    # [应用名称] 开发设计文档

## 1. 业务产出物  [第一步结论]
## 2. 能力拆解    [第二步结论]
## 3. 功能映射    [第三步结论]
## 4. 设计推理    [第四步结论]
## 5. 操作清单    [第五步执行记录]
## 6. 验收用例    [第六步验收用例]
## 7. 风险与待办  [已知风险 + 后续优化方向]
