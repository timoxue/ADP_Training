## 第一步：业务产出物

**核心问题**：我最终要交付什么？先定义交付物。

```
应用名称：投研小团队
目标用户：基金经理/研究员
核心场景：给一个股票代码，多线程并行完成——基本面分析、行业对比、新闻舆情、风险预警，最后汇总成投研快报
交付物形态：多Agent并行分析 → 汇总综合报告
  - 【财务分析】核心指标+同比变化+财务风险评估
  - 【行业对比】对标公司清单+关键指标对比+行业位置评估
  - 【舆情动态】最新新闻TOP5+深度文章TOP3+舆情风向
  - 【风险预警】风险信号清单(🔴🟡🟢)+关联交易+异常事件
  - 【综合结论】调度Agent整合判断
成功标准：比单人逐一查快3-5倍，输出结构完整可交付
```

### 为什么是"团队"而不是"助手"？

单个 Agent（如已有的"投研助手"）处理投研任务时是**串行**的：先查财务、再查行业、再查新闻、再查风险。一个完整的投研分析可能需要 4 轮对话才能覆盖全维度。

投研小团队的核心诉求是**并行**：用户给一个股票代码，4 个专业子 Agent 同时干活，主 Agent 汇总，一轮对话产出全维度投研快报。

### 与"投研助手"的定位区分

| 维度 | 投研助手（标准模式/Claw） | 投研小团队（Multi-Agent） |
|------|--------------------------|--------------------------|
| 核心价值 | 知识库+API 交叉校验，展示 AI 能力边界 | 多角色并行处理，模拟真实投研团队协作 |
| 处理方式 | 串行，一次查一个维度 | 并行，4 个维度同时分析 |
| 输出形态 | 按需触发（关键指标对照表/关联交易清单） | 固定结构投研快报（5 个模块齐全） |
| 适用场景 | 培训学员理解 AI 能力边界 | 基金经理/研究员快速获取全维度分析 |

!!! tip "培训要点"
    同一个培训课程里有两个投研相关应用，但定位完全不同。学员要能说清楚：什么情况下用投研助手，什么情况下用投研小团队。

## 第二步：能力拆解

**核心问题**：要产出它，需要哪几种能力？

| 优先级 | 能力 | 具体描述 | 对应子Agent |
|--------|------|---------|------------|
| P0 | 财务分析 | 从年报/半年报提取关键指标，计算毛利率/净利率/ROE等，同比环比变化 | 财务分析师 |
| P0 | 行业对比 | 同行业公司横向对比（营收/利润/估值），行业位置评估 | 行业分析师 |
| P0 | 新闻舆情 | 实时新闻+微信深度文章检索与摘要，舆情风向判断 | 舆情专员 |
| P0 | 风险预警 | 关联交易/诉讼/高管变动/现金流异常等风险信号识别，分级标注 | 风控专员 |
| P0 | 调度汇总 | 接收需求→任务拆解→分发→收齐→整合为投研快报 | 投研队长（主Agent） |
| P1 | 知识检索 | 从知识库（年报/研报/调研纪要）提取原文数据 | 财务/风控 |
| P1 | 数据可视化 | 生成指标对比图表 | 行业分析师 |
| P1 | 事实核查 | 舆情信息交叉验证 | 舆情专员 |

!!! tip "培训要点"
    **为什么"财务分析"和"行业对比"不合并？** 虽然两者都用 `a-stock-data`，但关注点不同：财务分析聚焦**公司自身**的指标变化（同比/环比/趋势），行业对比聚焦**公司之间**的横向定位（排名/分位/优劣势）。拆开后两个子 Agent 可以并行工作。

    **为什么"调度汇总"是 P0？** Multi-Agent 的核心难点不在子 Agent 的专业能力，而在主 Agent 的调度逻辑。如果调度 Agent 判断错误，整个流程就会跑偏。

## 第三步：功能映射

**核心问题**：每种能力对应平台哪个功能？选择哪种应用模式？

### 三步映射法

| 轮次 | 方法 | 发现的技能 |
|------|------|-----------|
| 第一轮：经验初筛 | 凭经验列出核心技能 | a-stock-data、news、wechat-article-search、adp-prompt-builder、adp-app-manager（5个） |
| 第二轮：平台搜索 | 在平台 Skill 目录搜索补充 | 补充发现：dcf-model、tencent-fact-check、summarize-cn、tencentcloud-adp-kb-search、data-analysis-workflow、svg-visualizer（6个） |
| 第三轮：匹配度排序 | 按子Agent职责筛选 | 最终选定11个技能+2个工具 |

### 最终映射结果

| 子Agent | 负责能力 | 挂载技能 | 挂载工具 |
|---------|---------|---------|---------|
| 财务分析师 | 财务分析 | a-stock-data、dcf-model | 知识库问答 |
| 行业分析师 | 行业对比 | a-stock-data、data-analysis-workflow、svg-visualizer | 知识库问答 |
| 舆情专员 | 新闻舆情 | news、wechat-article-search、tencent-fact-check、summarize-cn | — |
| 风控专员 | 风险预警 | a-stock-data、tencentcloud-adp-kb-search | 知识库问答 |
| 投研队长（主） | 调度汇总 | adp-prompt-builder（可选） | 知识库问答、图片理解 |

### 模式选择：Multi-Agent

**决策：Multi-Agent 模式**

| 维度 | 理由 |
|------|------|
| 不选 Claw | 单 Agent 无法真正并行，体现不出"团队"优势 |
| 不选标准模式 | 无多角色协作机制 |
| 不选单工作流 | 业务流程不固定，用户每次需求维度不同 |
| 选 Multi-Agent | 多角色并行是核心诉求，4 个子 Agent 同时干活，主 Agent 汇总，一轮对话产出全维度快报 |

!!! warning "与其他场景对比"
    投研助手选 Claw（灵活调用多技能）→ 财富顾问选标准模式（合规硬拦截+知识库稳定）→ 投研小团队选 Multi-Agent（多角色并行是核心）。同一个平台，三个不同场景选了三种不同模式。

## 第四步：设计推理

**核心问题**：为什么选它？边界与兜底在哪？

### 决策1：应用模式

```
能力：多角色并行处理
选它：Multi-Agent — 4个子Agent同时干活，主Agent汇总，一轮产出全维度快报
不选：Claw（单Agent串行）、标准模式（无多角色机制）
边界：调度Agent可能误判问题类型分错子Agent
兜底：转交规则用关键词互斥定义，子Agent输出格式标准化（Markdown标题+表格）
```

### 决策2：子Agent数量

```
能力：专业角色分工
选它：4个子Agent + 1个调度 = 5个
不选：更多（如拆出估值分析师）→ 调度复杂度太高
不选：更少（如合并财务+行业）→ 失去并行优势
边界：4个是Multi-Agent的合理上限
兜底：某个子Agent处理不了时转回调度Agent重新分配
```

### 决策3：转交规则

```
能力：任务路由
选它：关键词互斥分发
  - 财务分析师 ← 财务指标、利润表、ROE、毛利率、估值
  - 行业分析师 ← 行业对比、同业对标、竞争格局、市场份额
  - 舆情专员   ← 新闻、舆情、媒体观点、公众号文章
  - 风控专员   ← 关联交易、风险提示、解禁、减持、问询函
不选：自然语言模糊匹配 → 调度Agent可能误判
风险：用户提问跨多个领域时，需同时转交多个子Agent
兜底：提示词明确说明"一个需求可同时分发给多个子Agent"
```

### 决策4：数据源分配

```
能力：数据获取
选它：按子Agent分工分配数据源，提示词约束各子Agent只用自己领域的数据源
不选：所有Agent共享所有数据源 → 权限混乱，舆情专员不该查关联交易公告
边界：a-stock-data的skill_id只能挂载到整个应用，子Agent都能调用
兜底：提示词约束各子Agent只用自己领域的数据源（软约束）
```

### 决策5：合规治理

```
合规强度：🟡 中（内部投研场景）
选它：两层防线
  - 提示词约束（软）：每个子Agent提示词末尾写入"不承诺收益，不预测精确价格"
  - 免责声明（兜底）：主Agent汇总报告末尾固定附带"本内容由AI基于公开数据生成，仅供内部参考，不构成投资建议"
检出：提示词要求风控专员标注风险信号（🔴🟡🟢分级）
拦截：按需开启内容安全（Multi-Agent有硬拦截能力，内部场景按需配置）
脱敏：按需开启隐私脱敏
```

### 决策6：协作方式

```
能力：任务编排
选它：自动编排——调度Agent根据问题自动分发，无需用户手动选择子Agent
不选：手动指定 → 用户需要知道有哪些子Agent，增加认知负担
风险：调度Agent误判
兜底：提示词要求"如果用户需求不明确，先确认标的和分析维度再分发"
```

### 搭建模式推荐结论

```
📋 搭建模式推荐结论

推荐模式：Multi-Agent 模式
推理依据：
  1. 多角色并行是核心诉求——4个专业子Agent同时干活，主Agent汇总
  2. 无固定执行流程——用户每次需求维度不同，不适合单工作流
  3. 无合规硬拦截需求——内部投研场景，提示词软约束+免责声明足够

替代方案：Claw+提示词角色分级（用角色切换替代多Agent，但失去并行优势）
⚠️ 关键风险：调度复杂度高；子Agent输出格式不统一会导致汇总丢信息
🔒 合规风险：低（内部投研，软约束+免责声明）
```

    所有需要手动操作的步骤，**提示词和技能 ID 已全部准备好**，详见各配置页面。
