# LLM · RAG · Agent

> 理解三个核心概念：大语言模型是什么、检索增强生成解决什么问题、智能体比普通 AI 多了什么。

---

## 大语言模型（LLM）

### 从 2017 年的一篇论文说起

2017 年，Google 发表了一篇改变行业走向的论文，题目叫 **《Attention Is All You Need》**——注意力就是一切。

这篇论文提出了 **Transformer 架构**，核心是"注意力机制"：模型在处理文字时，会自动学会"该关注哪些词、忽略哪些词"。

就是这个机制，催生了后来的 ChatGPT、Claude、Gemini、DeepSeek 这一代大模型——把 AI 从"只会下棋"推向了"能对话、能推理、能写代码"。

### LLM 是怎么工作的

LLM 的本质是一个**超大规模的文字预测机器**。

训练阶段，它读遍了互联网上的海量文字，学会了"这句话后面最可能接什么"。推理阶段，它根据你的输入，一个字一个字地预测下一个最合适的词。

```
你输入："今天天气"
模型预测：很（概率 40%）/ 不（概率 25%）/ 真（概率 20%）...
选择"很" → 继续预测 → "好" → 继续预测 → "，"...
最终输出："今天天气很好，适合出门。"
```

这个过程看起来简单，但模型在"预测"的背后，已经理解了逻辑、语法、事实，甚至能做推理。

### 关键概念

| 概念 | 说明 |
|------|------|
| **Token（词元）** | 模型处理文字的基本单位，约等于半个汉字或一个英文单词。计费、限制都按 token 计算 |
| **上下文窗口** | 每次对话模型能"看到"的最大 token 数。超出后早期内容会被遗忘 |
| **温度（Temperature）** | 控制输出的随机性。0 = 确定性强，适合分析；1 = 创意性强，适合写作 |
| **知识截止日期** | 模型的训练数据有时效性，无法感知截止日后的信息 |
| **幻觉（Hallucination）** | 模型生成听起来合理但实际上错误的内容，是当前最大的应用风险 |

### LLM 的三大局限

很多人用了一段时间 AI 之后，都有类似的感受："挺好用的，但好像只能用来写写文案、改改文档。"

原因在这里——LLM 本质上是个"大脑"，但它：

- **没有手脚**：无法主动获取最新数据、打开文件、调用系统
- **没有记忆**：每次对话独立，不知道你上次说了什么
- **没有纪律**：无法稳定地执行多步任务，容易跑偏

就像把天才实习生关进玻璃房——对答如流，但拉不到最新行情，打不开你的文件夹，发不出去邮件。

**要让 AI 真正替你干活，就需要给它装上三样东西：手脚（工具调用）、记忆（上下文管理）、纪律（稳定执行）。** 这就是 RAG 和 Agent 要解决的问题。

## 检索增强生成（RAG）

### RAG 解决什么问题

LLM 不知道你公司的内部制度、不认识你的产品说明书、不知道昨天发布的监管通知。

RAG（Retrieval-Augmented Generation，检索增强生成）的思路是：**不让模型"记住"所有知识，而是在需要的时候去找**。

### RAG 的工作流程

```
用户提问："宁德时代 Q3 研报的核心观点是什么？"
        ↓
1. 把问题转换成向量（数字表示的"语义"）
        ↓
2. 在知识库里找语义最相近的文档片段
        ↓
3. 把找到的片段 + 用户问题一起发给 LLM
        ↓
4. LLM 基于真实材料生成回答，标注出处
```

### RAG vs 微调（Fine-tuning）

两种方式都能让模型更了解你的业务，但适合不同情况：

| 对比维度 | RAG | 微调 |
|---------|-----|------|
| 适合的知识类型 | 随时更新的文档、大量资料 | 固定风格、特定格式、专业术语 |
| 更新成本 | 低，上传新文档即可 | 高，需要重新训练 |
| 可追溯性 | 高，能标注来源 | 低，知识融入模型权重 |
| 建议场景 | 企业知识库、研报问答 | 客服话术固化、特定行业语言风格 |

**证券场景基本都用 RAG**：研报、公告、产品说明书更新频繁，且监管要求能追溯信息来源，RAG 完全符合需求。

### 影响 RAG 质量的关键因素

- **分块策略**：文档切成多大的片段？太大检索不精准，太小上下文丢失
- **Top-K**：每次检索返回几个片段？建议 5–10
- **相似度阈值**：低于多少分的片段不返回？建议 0.5–0.7
- **混合检索**：同时用语义检索（向量）和关键词检索（BM25），取长补短
- **重排序（Rerank）**：检索结果二次排序，把最相关的放最前面

!!! warning "RAG 不是万能的"
    RAG 解决"找到正确材料"的问题，但如果材料本身有错误，或者问题需要跨多个文档综合推理，RAG 的效果会下降。这时需要更精细的工作流设计。

## 智能体（Agent）

### Agent 与 LLM 的本质区别

| | LLM（纯对话） | Agent |
|--|------------|-------|
| 模式 | 一问一答，每轮独立 | 持续执行，多步完成任务 |
| 工具 | 无 | 可调用搜索、数据库、代码执行等 |
| 记忆 | 单次上下文 | 可有外部记忆 |
| 目标 | 回答问题 | 完成任务 |

### Agent 的核心循环

Agent 的本质是一个 **while 循环**：

```
接收任务
    ↓
┌─────────────────────────────┐
│  思考：当前状态是什么？       │
│  决策：下一步该做什么？       │
│  行动：调用工具/生成内容      │
│  观察：结果是什么？           │
│  是否完成？→ 否 → 继续循环   │
└─────────────────────────────┘
    ↓（完成）
输出结果
```

以前用 AI，你是一问一答，每次交互是独立的。Agent 把"注意力"从单次交互，变成了对整个任务状态的持续追踪——它一直知道自己在干什么、干到哪一步了、还有什么没做完。

### Agent 可以调用的工具类型

| 工具类型 | 典型能力 | 证券场景举例 |
|---------|---------|------------|
| 搜索工具 | 网页搜索、新闻检索 | 抓取监管公告 |
| 数据工具 | API 查询、数据库查询 | 获取实时行情、财务数据 |
| 代码执行 | 运行 Python/JS | 计算估值模型、生成图表 |
| 文件操作 | 读写文档、解析 PDF | 处理研报、合同 |
| 通知工具 | 发邮件、推送消息 | 推送简报到企业微信 |

### Agent 的记忆类型

| 记忆类型 | 说明 | 持久性 |
|---------|------|--------|
| 上下文记忆 | 当前会话的对话历史 | 会话结束即消失 |
| 外部存储 | 向量数据库、文件系统 | 持久化 |
| 长期记忆 | 用户画像、历史偏好摘要 | 持久化，跨会话 |

### Agent 的三个"暗面"

Agent 能力越强，风险也越大。使用前必须了解三类常见问题：

**1. 自我欺骗**

Agent 会"自信地编"——把猜测当事实交付，语气笃定，内容却是虚构的。在金融场景，这是致命风险。

**2. 记忆污染**

如果错误信息进了长期记忆，它会一错再错——而且越用越坚信那个错误结论。

**3. 目标漂移**

在多步任务里，Agent 可能跑着跑着就忘了最初要干什么，干着干着就偏题了。

!!! danger "金融场景必须做治理"
    这三个问题的本质都是"注意力跑偏了"。金融场景必须有**人工审核节点**，必须**留痕可审计**。这不是负担，是底线。

### Multi-Agent：多个 Agent 协作

当任务足够复杂（比如对一家公司做全方位尽职调查），单个 Agent 搞不定，就需要多个 Agent 分工协作：

```
主 Agent（调度）
├── 财务分析 Agent  → 处理年报、财务三表
├── 行业研究 Agent  → 处理研报、行业数据
├── 舆情监控 Agent  → 处理新闻、公告
└── 合规审查 Agent  → 检查风险提示
```

关键是**转交关系**的配置——主 Agent 需要知道什么情况下把任务交给哪个子 Agent。

## 三者关系总结

```
LLM  ——  思考与生成的引擎
  +
RAG  ——  接入企业私有知识，解决幻觉和时效问题
  +
Agent  ——  赋予行动能力，让 AI 真正干活
  ‖
智能体应用 ——  能理解、能检索、能执行的数字员工
```

**一句话记住核心逻辑**：
> 大模型足够聪明，但被关在玻璃房里。RAG 给它装上了"图书馆"，Agent 给它装上了"手脚"，再加上"纪律"（稳定的执行流程），才能真正替你干活。
