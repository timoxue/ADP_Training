> 单工作流模式 | 标签提取 → 条件判断 → 风险清单 | app_id: 2068701020736650752

## 二、整体架构

### 核心设计：标签提取 → 条件判断 → 风险清单

```
┌──────────────────────────────────────────────────────────────────────────┐
│  主工作流 compliance_precheck（多文件批量预检）                              │
│                                                                          │
│  [开始] → [文件收集(5个)] → [逐文件预检(ITERATION)] → [汇总输出] → [结束]    │
│                                         │                                │
│                                         │ 每个文件迭代一次                  │
│                                         ▼                                │
│  ┌───────────────────────────────────────────────────────────────────┐   │
│  │  子工作流 compliance_single_check（单文件预检 + 条件分支）            │   │
│  │                                                                   │   │
│  │  [开始] → [文档解析] → [LLM分析] → [标签提取]                      │   │
│  │                                              │                    │   │
│  │                                    [条件判断(LOGIC_EVALUATOR)]     │   │
│  │                                     /        |        \            │   │
│  │                               高风险      中风险      低风险         │   │
│  │                                 ↓          ↓          ↓          │   │
│  │                            [拦截输出]  [警告输出]  [通过输出]        │   │
│  │                                 \          |          /            │   │
│  │                                  → [结束] ←                       │   │
│  └───────────────────────────────────────────────────────────────────┘   │
└──────────────────────────────────────────────────────────────────────────┘
```

### 工作原理

1. 用户上传多个文件（最多5个）
2. FILE_COLLECTION收集所有文件，输出FileList数组
3. ITERATION节点遍历FileList，每次取1个文件传给子工作流
4. 子工作流对每个文件执行：
   - DocParse解析文件内容
   - LLM合规分析（检出违规、评估风险等级）
   - PARAM_EXT标签提取（结构化提取6个参数）
   - **LOGIC_EVALUATOR条件判断**（按risk_level分流）
   - 高风险 → 拦截输出（阻断+必须修改）
   - 中风险 → 警告输出（有条件通过+建议）
   - 低风险 → 通过输出（通过+优化建议）
5. 主工作流ANSWER汇总所有迭代结果，输出批量预检报告

## 四、Phase 1：搭建子工作流 compliance_single_check

!!! info "为什么先建子工作流"
    子工作流是单文件预检的完整流程，每次处理1个文件，含条件分支。必须先建子工作流，因为主工作流的ITERATION节点需要引用它。

### 步骤1：创建子工作流

**操作路径**：ADP控制台 → 应用管理 → 合规预检助手 → 工作流管理 → 新建工作流

| 字段 | 填写内容 |
|---|---|
| 工作流名称 | `compliance_single_check` |
| 描述 | 逐文件合规预检-子工作流（条件分支） |

点击「确定」→ 进入画布编辑器。

### 步骤2：搭建9节点分支流

从左侧节点面板依次拖入9个节点：

```
[开始] → [文档解析] → [合同风险分析] → [标签提取] → [条件判断]
                                                        │
                                            ┌───────────┼───────────┐
                                            ▼           ▼           ▼
                                        [拦截输出]  [警告输出]  [通过输出]
                                            └───────────┼───────────┘
                                                        ▼
                                                      [结束]
```

**主干连线**（从节点右侧拖到下一节点左侧）：

1. [开始] → [文档解析]
2. [文档解析] → [合同风险分析]
3. [合同风险分析] → [标签提取]
4. [标签提取] → [条件判断]

**分支连线**（从条件判断节点拖出3条线）：

5. [条件判断] → [拦截输出]（高风险分支）
6. [条件判断] → [警告输出]（中风险分支）
7. [条件判断] → [通过输出]（低风险分支）

**汇合连线**：

8. [拦截输出] → [结束]
9. [警告输出] → [结束]
10. [通过输出] → [结束]

!!! warning "连线注意"
    条件判断节点拖出3条线时，每条线对应一个分支条件。在LOGIC_EVALUATOR配置中需要为每条线设置条件表达式。

### 步骤4：配置「文档解析」插件节点

这个节点的作用是把上传的文件解析成文本，让LLM能读到文件内容。

**4.1 选择插件**

点击「文档解析」节点 → 右侧配置面板 → 点击「选择插件」→ 搜索 `DocParse` → 选择该插件。

| 配置项 | 值 |
|---|---|
| 插件名称 | DocParse |
| ToolID | 44b4c776-b1a4-4b3a-a2dc-7355c1ad193b |
| 支持格式 | PDF, DOC, DOCX, PPT, PPTX, TXT, XLSX, CSV |

**4.2 插件输出说明**

DocParse解析文件后输出3个字段：

| 输出字段 | 类型 | 说明 |
|---|---|---|
| Output.Code | INT | 返回码。0=正常，非0=异常 |
| Output.Msg | STRING | 返回信息 |

**5.3 系统提示词(SystemPrompt)**

```
你是「合规预检助手」，专门为证券行业营销话术和合同文档提供合规预检服务。你的职责是检出违规表述、评估风险等级、给出合规建议。你必须严格遵守三条红线：不承诺收益、不代客操作、不构成投资建议。
```

**5.4 主提示词(Prompt)**

```
## 三条不可逾越的红线
1. ❌ 不承诺收益 — 不得出现保本/保收益/稳赚/无风险等表述
2. ❌ 不代客操作 — 不得出现帮你买入/替你操作/代你交易等表述
3. ❌ 不构成投资建议 — 不得出现强烈推荐/必买/必涨/闭眼买等表述

## 预检流程
1. 判断输入类型：marketing(营销话术) / contract(合同) / other(其他)
2. 营销话术：逐项检查是否命中以下6类违规：
   - 保本承诺（保本/保收益/稳赚/无风险/零风险/保证盈利）
   - 代客操作（帮你买入/替你操作/代你交易/我来操盘）
   - 投资建议诱导（强烈推荐/必买/必涨/闭眼买/赶紧加仓）
   - 误导性表述（高收益/秒到账/零门槛/内部消息/庄家拉升）
   - 不当背书（证监会推荐/监管背书/政府扶持）
   - 业绩展示不规范（年化X%无来源/历史最佳业绩）
3. 合同：检查是否包含以下关键条款：
   - 风险提示条款（是否充分披露风险）
   - 适当性评估条款（是否匹配投资者风险等级）
   - 代客操作禁止条款（是否明确禁止代客操作）
   - 收益禁止承诺条款（是否明确禁止承诺收益）
   - 如含违规条款（如隐含收益承诺），标记为高风险
4. 如为"other"或超出预检范围（如用户要求投资建议），标记is_redline=true，risk_level=high

## 风险等级判定标准
- high（高风险）：命中红线词（保本/保收益/稳赚/代客操作/投资建议诱导），或命中2个以上高风险词
- medium（中风险）：命中1个高风险词但非红线，或命中2个以上中风险词（误导性表述/不当背书）
- low（低风险）：仅命中低风险词（模糊表述/措辞建议），或无违规项

## 输出要求
对输入内容进行分析，输出以下信息：
- 输入类型判定及理由
- 每个检出违规项：原文片段、命中关键词、违规类型、具体法规名+条款号
- 风险等级判定及理由（必须明确输出 risk_level=high/medium/low）
- 合规替换建议
- 引用的法规条文（必须使用证券行业专项法规：《证券法》《金融营销通知》《证券投资顾问暂行规定》《广告法》《适当性管理办法》等）

---以下为输入内容---

文件内容：
{{doc_content}}

用户输入文本：
{{user_input}}
```

!!! tip "变量名一致性"
    Prompt中的 `{{doc_content}}` 和 `{{user_input}}` 必须和输入变量名完全一致。注意新增了"风险等级判定标准"部分，确保LLM输出的risk_level值标准化。

### 步骤6：配置「标签提取(PARAMETER_EXTRACTOR)」节点

这个节点从LLM的分析文本中结构化提取6个关键参数，其中 `risk_level` 将作为条件判断的依据。

**6.1 基础配置**

| 参数 | 填写值 | 原因 |
|---|---|---|
| 模型(ModelName) | `Youtu/youtu-intent-pro` | 专为参数提取优化 |
| Temperature | `0` | 精确提取，不允许随机 |

**6.2 输入变量**

| 变量名 | 引用类型 | 引用节点 | 引用字段 |
|---|---|---|---|
| `user_input` | 引用节点输出 | 合同风险分析 | Output.Content |

> LLM的输出在 Output.Content 中，PARAM_EXT从这里读取分析结果文本。

**6.3 提取提示词(UserConstraint)**

```
请从以下分析结果中精确提取参数：{{user_input}}

提取规则：
- input_type：输入类型，严格取值 marketing/contract/other
- risk_level：风险等级，严格取值 high/medium/low（这是条件判断的关键字段，必须准确）
- violations：每项格式为"原文片段→违规类型→法规依据"，多项用分号分隔
- suggestions：每条建议用分号分隔
- law_references：每项法规格式为"《法规名》第X条"，多项用分号分隔
- is_redline：是否触碰三条红线（承诺收益/代客操作/构成投资建议），严格取值 true/false
- file_name：当前文件名，直接透传
```

**6.4 参数定义（7个）**

在「参数定义」区域逐个添加以下参数：

| # | 参数名 | 类型 | 必填 | 描述（≤30字符） |
|---|---|---|---|---|
| 1 | `input_type` | STRING | 是 | 类型：marketing/contract/other |
| 2 | `risk_level` | STRING | 是 | 等级：high/medium/low |
| 3 | `violations` | STRING | 否 | 检出违规项：关键词+类型+法规 |
| 4 | `suggestions` | STRING | 否 | 针对违规项的修改建议 |
| 5 | `law_references` | STRING | 否 | 引用法规：名称+条款号 |
| 6 | `is_redline` | STRING | 是 | 是否触红线：true/false |
| 7 | `file_name` | STRING | 否 | 当前文件名（透传） |

**操作方式**：每次点击「新增参数」→ 填写参数名、类型、描述 → 点「确定」→ 重复7次。

!!! warning "描述字段长度限制"
    描述字段单个不超过30个字符，否则会报错。

**6.5 输出定义（7个）**

!!! danger "最容易踩的坑"
    这是整个工作流最容易出错的地方。参数定义了7个，输出定义也必须有7个，且字段名完全一致。如果输出定义只有默认的 `keyword` 一个字段，下游LOGIC_EVALUATOR和ANSWER引用 `Output.risk_level` 等时会报错。

在「输出定义」区域逐个添加以下输出字段：

| # | 输出字段名 | 类型 | 说明 |
|---|---|---|---|
| 1 | `input_type` | STRING | 输入类型 |
| 2 | `risk_level` | STRING | 风险等级（条件判断依据） |
| 3 | `violations` | STRING | 违规项 |
| 4 | `suggestions` | STRING | 修改建议 |
| 5 | `law_references` | STRING | 法规引用 |
| 6 | `is_redline` | STRING | 是否触红线 |
| 7 | `file_name` | STRING | 文件名 |

**6.6 数据流对齐验证**

```
LLM输出                     PARAM_EXT输入              PARAM_EXT参数          PARAM_EXT输出
Output.Content ──────────→ user_input            → (模型提取)          → input_type (STRING)
                                                   → (模型提取)          → risk_level (STRING) ← 条件判断依据
                                                   → (模型提取)          → violations (STRING)
                                                   → (模型提取)          → suggestions (STRING)
                                                   → (模型提取)          → law_references (STRING)
                                                   → (模型提取)          → is_redline (STRING)
                                                   → (透传)              → file_name (STRING)
```

### 步骤7：配置「条件判断(LOGIC_EVALUATOR)」节点

这个节点读取PARAM_EXT提取的 `risk_level` 值，按高/中/低三路分流到不同的ANSWER节点。

**7.1 输入变量**

| 变量名 | 引用类型 | 引用节点 | 引用字段 |
|---|---|---|---|
| `risk_level` | 引用节点输出 | 标签提取 | Output.risk_level |

**7.2 条件分支配置**

配置3个条件分支：

=== "分支1：高风险"

    | 配置项 | 填写值 |
    |---|---|
    | 条件名称 | 高风险 |
    | 左值(Left) | risk_level |
    | 运算符(Operator) | EQ（等于） |
    | 右值(Right) | `high` |
    | 下一节点 | 拦截输出 |

=== "分支2：中风险"

    | 配置项 | 填写值 |
    |---|---|
    | 条件名称 | 中风险 |
    | 左值(Left) | risk_level |
    | 运算符(Operator) | EQ（等于） |
    | 右值(Right) | `medium` |
    | 下一节点 | 警告输出 |

=== "分支3：低风险"

    | 配置项 | 填写值 |
    |---|---|
    | 条件名称 | 低风险 |
    | 左值(Left) | risk_level |
    | 运算符(Operator) | EQ（等于） |
    | 右值(Right) | `low` |
    | 下一节点 | 通过输出 |

**7.3 工作原理**

```
标签提取输出 risk_level = "high"
        ↓
LOGIC_EVALUATOR 判断：
  risk_level EQ "high"   → ✅ 匹配 → 走「拦截输出」分支
  risk_level EQ "medium" → ❌ 不匹配
  risk_level EQ "low"    → ❌ 不匹配
```

!!! warning "大小写敏感"
    每个条件的「目标节点」必须和步骤2的连线一致。right值必须是小写：`high` / `medium` / `low`。

### 步骤8：配置3个分支ANSWER节点

3个ANSWER节点的**输入变量完全相同**（都引用标签提取的7个输出字段），只是回复模板不同。

=== "拦截输出（高风险🚫）"

    **输入变量**：

    | 变量名 | 引用节点 | 引用字段 |
    |---|---|---|
    | `file_name` | 标签提取 | Output.file_name |
    | `input_type` | 标签提取 | Output.input_type |
    | `violations` | 标签提取 | Output.violations |
    | `suggestions` | 标签提取 | Output.suggestions |
    | `law_references` | 标签提取 | Output.law_references |
    | `is_redline` | 标签提取 | Output.is_redline |

    **回复模板**：

    ```
    🚫 {{file_name}} — 预检不通过（高风险）

    【输入类型】{{input_type}}
    【是否触碰红线】{{is_redline}}

    【检出违规项】
    {{violations}}

    【必须修改建议】
    {{suggestions}}

    【法规依据】
    {{law_references}}

    ❌ 处置结论：本内容存在高风险违规，已被拦截。必须修改上述违规内容后重新提交预检。
    ```

## 五、Phase 2：搭建主工作流 compliance_precheck

!!! info "前提"
    Phase 1子工作流已调试通过（高风险走拦截分支、低风险走通过分支），WorkflowID已记录。

### 步骤1：进入主工作流画布

**操作路径**：ADP控制台 → 应用管理 → 合规预检助手 → 工作流管理 → `compliance_precheck` → 画布编辑器

### 步骤2：清理旧节点

删除以下节点（右键 → 删除）：

- ❌ 文档解析（PLUGIN）— 如有
- ❌ 合同风险分析（LLM）— 如有
- ❌ 标签提取（PARAMETER_EXTRACTOR）— 如有
- ❌ 条件判断（LOGIC_EVALUATOR）— 如有
- ❌ 拦截输出/警告输出/通过输出（ANSWER）— 如有

保留的节点：✅ 开始 / ✅ 合同文件收集 / ✅ 结束

### 步骤3：修改FILE_COLLECTION节点

| 配置项 | 旧值 | 新值 | 原因 |
|---|---|---|---|
| MaxAllowedUploadFileCount | 1 | **5** | 支持批量上传多个文件 |

### 步骤4：新增并连线

### 步骤5：配置ITERATION节点

**基础配置**：

**输入变量**：

| 变量名 | 类型 | 引用节点 | 引用字段 |

**遍历配置**：

| 配置项 | 填写值 |
|---|---|

| 子工作流参数 | 映射来源 | 说明 |
|---|---|---|
| `file_url` | `file_list.Item.FileURL` | 每次取当前文件的URL |

!!! warning "参数名必须精确匹配"
    映射的3个参数名必须和子工作流「开始」节点定义的工作流参数名完全一致。

## 附录B：条件分支设计说明

### 为什么要用条件分支

| 维度 | 说明 |
|---|---|
| 处置精度 | 高/中/低风险分别用不同模板，不同处置语气 |
| 高风险处置 | 🚫拦截+明确阻断提示+必须修改后重新提交 |
| 中风险处置 | ⚠️警告+有条件通过+需合规部门复核 |
| 低风险处置 | ✅通过+优化建议为可选 |
| 合规要求 | 检出（PARAM_EXT）和处置（分支ANSWER）分离 |

### 三条红线与风险等级对应

| 红线 | 典型词 | risk_level | 走哪个分支 |
|---|---|---|---|
| ❌ 不承诺收益 | 保本/保收益/稳赚/无风险 | high | 🚫 拦截输出 |
| ❌ 不代客操作 | 帮你买入/替你操作/代你交易 | high | 🚫 拦截输出 |
| ❌ 不构成投资建议 | 强烈推荐/必买/必涨/闭眼买 | high | 🚫 拦截输出 |
| — | 命中1个高风险词但非红线 | medium | ⚠️ 警告输出 |
| — | 仅命中低风险词或无违规 | low | ✅ 通过输出 |
