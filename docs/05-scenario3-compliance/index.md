# 场景三：合规风控 — 六步推理链分析

> **把"逐字核对"这种最熬人的活做成自动化流水线**

---

## 方案选择

| | 方案A：话术合规预检 | 方案B：制度版本比对 |
|---|---|---|
| **输入** | 营销话术/合同文本（批量上传，最多5个） | 旧版+新版制度文档（各1份） |
| **输出** | 风险等级清单（高/中/低 + 违规项 + 处置建议） | 差异比对清单（新增/删除/修改 + 影响程度） |
| **核心设计** | ITERATION批量遍历 + LOGIC_EVALUATOR条件分支 | 双FILE_COLLECTION并行 + LLM逐条比对 |
| **何时选** | 日常话术审核、批量合规检查 | 制度更新发布后、合同版本对比 |

---

## 方案A：话术合规预检 六步分析

### 第一步：业务产出物

| 项目 | 内容 |
|---|---|
| 应用名称 | 合规预检助手 |
| 目标用户 | 证券公司合规部门、营销部门 |
| 核心场景 | 营销话术合规检查、合同条款合规审核 |
| 输入 | 营销话术文本或合同文档（文件上传） |
| 交付物 | 风险预检清单（风险等级 + 违规项 + 处置建议 + 法规依据） |
| 成功标准 | "保本保息"→ 🚫拦截；"收益稳健"→ ⚠️警告；"投资有风险"→ ✅通过 |

### 第二步：能力拆解

| 能力 | 优先级 | 说明 |
|---|---|---|
| 知识检索 | P0 | 检索合规规则、禁语词表、法规条文 |
| 推理分析 | P0 | 检出违规表述、评估风险等级 |
| 合规风控 | P0 | 三条红线判定、风险分级（高/中/低） |
| 内容生成 | P1 | 按模板输出预检报告 |
| 多文件处理 | P1 | 批量预检多个文件 |

### 第三步：功能映射 + 模式选择

| 能力 | 最终映射 |
|---|---|
| 知识检索 | `tencentcloud-adp-kb-management`（合规规则库/禁语词表/合同模板/历史案例） |
| 推理分析 | LLM节点（deepseek-v3.2）+ 提示词三条红线判定标准 |
| 合规风控 | LOGIC_EVALUATOR（risk_level分支）+ 平台内容安全/保守回答/隐私脱敏 |
| 多文件处理 | ITERATION迭代节点（子工作流 + 主工作流） |
| 内容生成 | 3个差异化ANSWER节点（🚫/⚠️/✅） |

**模式选择：单工作流模式**

| 候选 | 结论 |
|---|---|
| Claw模式 | ❌ 无平台级合规硬拦截，不符合合规场景要求 |
| **单工作流模式** | ✅ 固定流程 + 完整安全设置（内容安全/保守回答/隐私脱敏） |
| 标准模式 | ⚠️ 可选，但工作流非强制，灵活度不够 |
| Multi-Agent | ❌ 单角色足够，调度复杂度不必要 |

### 第四步：设计推理

**决策1：风险分级 → 条件分支处置**

| 维度 | 选择 | 理由 |
|---|---|---|
| 选它 | LOGIC_EVALUATOR 3分支 | 高/中/低风险处置语气完全不同，单一模板精度不够 |
| 不选 | 单一ANSWER + risk_level变量 | 输出语气无差异，无法满足"高风险必须阻断"要求 |
| 边界 | 仅支持枚举值匹配（EQ high/medium/low） | 不支持数值范围比较 |
| 兜底 | LLM Prompt明确要求只输出"high/medium/low"三个值 | 避免LOGIC_EVALUATOR无分支命中 |

**决策2：单文件子工作流 + 主工作流ITERATION**

| 维度 | 选择 | 理由 |
|---|---|---|
| 选它 | 子工作流 + ITERATION | 每个文件独立预检，结果独立，汇总完整 |
| 不选 | 单一工作流处理多文件 | 无法逐文件分支判断，条件分支只能判断最后一个文件 |
| 边界 | ITERATION最多遍历5个文件 | MaxFiles=5 |
| 兜底 | 子工作流失败→ITERATION跳过当前文件继续 | 不影响其他文件预检 |

**搭建模式推荐结论**

```
📋 推荐模式：单工作流模式

推理依据：
  1. 合规场景必须有平台级硬拦截 → 不能选Claw模式
  2. 流程固定（收集→解析→分析→提取→判断→输出）→ 不需要Multi-Agent
  3. 需要LOGIC_EVALUATOR条件分支 → 单工作流原生支持
  4. 批量文件处理 → ITERATION迭代

⚠️ 关键风险：平台安全设置（内容安全/保守回答/隐私脱敏）必须手动开启，无法自动配置
🔒 合规风险：高（金融合规场景，三条红线必须配置，提示词+平台硬拦截双保险）
```

### 第五步：分步操作（概要）

> 详细分步配置见 → [**方案A完整搭建指南**](plan-a-speech.md)

| 步骤 | 做什么 | 关键配置 |
|---|---|---|
| 1 | 子工作流开始节点 | 3个WorkflowParams：file_name / file_url / user_input |
| 2 | 子工作流DocParse | ToolID: `44b4c776-b1a4-4b3a-a2dc-7355c1ad193b` |
| 3 | 子工作流LLM | deepseek-v3.2, Temp: 0.1，含风险等级判定Prompt |
| 4 | 子工作流PARAM_EXT | 7个参数 + 7个输出字段定义 |
| 5 | 子工作流LOGIC_EVALUATOR | 3分支：risk_level EQ high / medium / low |
| 6 | 3个ANSWER节点 | 🚫拦截 / ⚠️警告 / ✅通过，差异化模板 |
| 7 | 主工作流FILE_COLLECTION | MaxFiles=5 |
| 8 | 主工作流ITERATION | BodyType=WORKFLOW，遍历file_list，映射3个参数 |

### 第六步：验收

| 验收项 | 输入 | 通过标准 |
|---|---|---|
| 高风险拦截 | "保本保息稳赚不赔" | 🚫 risk_level=high，必须修改后重新提交 |
| 中风险警告 | "收益稳健适合所有人" | ⚠️ risk_level=medium，需合规复核 |
| 低风险通过 | "投资有风险入市需谨慎" | ✅ risk_level=low，预检通过 |
| 多文件批量 | 上传3个文件 | ITERATION遍历3次，汇总3份结果 |
| 红线触发 | "帮你买入XX股票" | is_redline=true，走🚫拦截分支 |
| 免责声明 | 任意输入 | 输出末尾含免责声明 |

## 方案B：制度版本比对 六步分析

### 第一步：业务产出物

| 项目 | 内容 |
|---|---|
| 应用名称 | 制度版本比对助手 |
| 目标用户 | 证券公司合规部门、制度管理部门 |
| 核心场景 | 制度文档新旧版本差异比对、合同版本对比 |
| 输入 | 旧版制度文档 + 新版制度文档（2个文件） |
| 交付物 | 差异清单（新增/删除/修改条款 + 影响程度 + 变更统计） |
| 成功标准 | 旧版3章 vs 新版4章 → diff_count正确，major/minor/editorial分类准确 |

### 第二步：能力拆解

| 能力 | 优先级 | 说明 |
|---|---|---|
| 文档解析 | P0 | 解析旧版和新版文档为文本 |
| 推理分析 | P0 | 逐条比对差异、判定影响程度 |
| 内容生成 | P0 | 按模板输出差异清单 |
| 知识检索 | P1 | 引用合规规则解释变更依据 |

### 第三步：功能映射 + 模式选择

| 能力 | 最终映射 |
|---|---|
| 文档解析 | DocParse插件 × 2（旧版实例 + 新版实例） |
| 推理分析 | LLM节点（deepseek-v3.2, Temp:0）同时接收两份文档 |
| 内容生成 | PARAM_EXT（8参数）+ ANSWER模板 |
| 知识检索 | `tencentcloud-adp-kb-management`（法规依据引用） |

**模式选择：单工作流模式**（同方案A，理由相同）

### 第四步：设计推理

**决策1：双文件处理方案**

| 维度 | 选择 | 理由 |
|---|---|---|
| 选它 | 双FILE_COLLECTION（各MaxFiles=1） | 两个文件分别命名，路由清晰可靠 |
| 不选 | 单FILE_COLLECTION + 索引分流 | 平台不支持用索引将FileList[0]/[1]分别路由到不同DocParse |
| 边界 | 每次只能上传1份旧版+1份新版 | 不支持批量多版本比对 |
| 兜底 | 对话提示明确区分"请上传旧版"/"请上传新版" | 避免用户上传顺序混淆 |

**决策2：差异三级影响程度**

| 维度 | 选择 | 理由 |
|---|---|---|
| 选它 | major / minor / editorial 三级 | 合规决策需知道哪些是"必须关注"的重大变更 |
| 不选 | 仅列出差异不分级 | 可读性差，100条差异无法区分轻重 |
| 边界 | 影响程度判断依赖LLM推理 | 可能出现分级偏差 |
| 兜底 | Prompt中给出三级定义和示例 | 减少LLM判断偏差 |

**搭建模式推荐结论**

```
📋 推荐模式：单工作流模式

推理依据：
  1. 流程固定（收集→解析→比对→提取→输出）→ 单工作流最合适
  2. 双文件并行处理，LLM同时接收两份文档 → 工作流节点天然支持
  3. 制度管理场景有合规要求 → 不选Claw

⚠️ 关键风险：LLM MaxTokens需设为8192以上，否则大文档比对输出截断
🔒 合规风险：中（内部工具，但制度文档属于内部敏感文档，建议开启隐私脱敏）
```

### 第五步：分步操作（概要）

> 详细分步配置见 → [**方案B完整搭建指南**](plan-b-diff.md)

| 步骤 | 做什么 | 关键配置 |
|---|---|---|
| 1 | 创建工作流 | 名称：regulation_diff_check |
| 2 | 旧版FILE_COLLECTION | MaxFiles=1，Question="请上传旧版制度文档" |
| 3 | 新版FILE_COLLECTION | MaxFiles=1，Question="请上传新版制度文档" |
| 4 | 两个DocParse | ToolID: `44b4c776-b1a4-4b3a-a2dc-7355c1ad193b`，分别连接旧版/新版 |
| 5 | LLM差异分析 | deepseek-v3.2, Temp:0, MaxTokens:8192，2个输入变量(old_content+new_content) |
| 6 | PARAM_EXT差异提取 | 8个参数 + 8个输出字段定义 |
| 7 | ANSWER差异清单 | 引用8个变量，含变更统计 + 分类清单 + 免责声明 |

### 第六步：验收

| 验收项 | 输入 | 通过标准 |
|---|---|---|
| 基础差异比对 | 旧版3章 vs 新版4章（增1章改3条） | diff_count=6, major_count=3, 新增条款含第4章 |
| 大文档比对 | 上传10页以上文档 | LLM输出完整，未被截断 |
| 无差异比对 | 两份相同文档 | diff_count=0，summary="两版文档内容一致" |
| 整章节删除 | 旧版5章 vs 新版4章（删1章） | deleted_items含被删除章节 |
| 三级影响程度 | 含major/minor/editorial各至少1项 | 统计正确，分类准确 |
| 免责声明 | 任意输入 | 输出末尾含免责声明 |
