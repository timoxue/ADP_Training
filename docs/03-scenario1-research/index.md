# 场景一：投研自动化 — 六步推理链分析

> **应用**：投研助手（app_id: 2067995783042319808）  
> **用途**：本页为培训参考答案，学员对照学习六步分析法如何在投研场景落地

---

## 六步推理链总览

```
业务产出物 → 能力拆解 → 功能映射 → 设计推理 → 分步操作 → 验收
   ↑_________________________________________↓
              （验收不过→回溯到对应步骤）
```

> 核心理念：每一步的输出是下一步的输入。跳步 = 返工。

---

## 第一步：业务产出物

**核心问题**：我最终要交付什么？先定义交付物。

```
应用名称：投研助手
目标用户：券商研究员/投研助理（培训学员）
核心场景：分析上市公司财务数据、提取关联交易、跨文档对比、检索研报公告
交付物形态：
  1. 《行业公司关键指标对照表》— 7项财务指标 × 多家公司，知识库+API交叉校验
  2. 《关联交易提取清单》— 关联方/交易类别/金额/定价/合规性，跨4类文档提取
  3. 实时资讯补充 — 财经新闻 + 公众号分析文章
成功标准：
  - 学员问"提取关联交易"→ Agent 命中4+份文档，输出结构化表格，每行标出处
  - 学员问"对比关键指标"→ Agent 知识库提取 + API交叉校验，不一致项⚠️标注
  - 学员问知识库没有的公司（如隆基绿能）→ Agent 在线补取，而非回答"不知道"
```

!!! tip "培训要点"
    交付物不是"一个聊天机器人"，而是**两个具体的结构化产出物 + 一个兜底能力**。先想清楚交付物长什么样，再倒推需要什么能力。

---

## 第二步：能力拆解

**核心问题**：要产出它，需要哪几种能力？

| # | 能力类别 | 具体描述 | 优先级 |
|---|---------|---------|--------|
| 1 | 知识检索 | 从年报/招股书/公告中提取财务数据和关联交易信息 | P0 |
| 2 | 数据查询 | 调用A股财务三表、巨潮公告、实时新闻等外部数据源 | P0 |
| 3 | 推理分析 | 知识库原文数据 vs API结构化数据交叉校验，标注差异 | P0 |
| 4 | 内容生成 | 按表格模板输出对照表/清单，每行标出处 | P0 |
| 5 | 合规风控 | 标注监管依据（深交所第23号公告格式），不承诺收益 | P1 |
| 6 | 在线补取 | 知识库未命中时，从巨潮/news/wechat在线补取 | P1（进阶版升为P0） |

!!! tip "培训要点"
    P0/P1的划分决定了第三步功能映射的优先级。**基础版只做P0，进阶版把P1的"在线补取"升为P0**——这就是两份培训文档的差异根源。

---

## 第三步：功能映射 + 模式选择

**核心问题**：每种能力对应平台哪个功能？选择哪种应用模式？

### 能力映射

| 能力 | 经验初筛 | 平台搜索验证 | 最终选择 | 匹配度 |
|------|---------|------------|---------|--------|
| 知识检索 | tencentcloud-adp-kb-management | tencentcloud-adp-kb-search、tencentcloud-adp-rerank | 知识库（平台原生）+ kb-management（文档管理） | ★★★ |
| 数据查询 | a-stock-data | news、wechat-article-search、dcf-model | a-stock-data（核心）+ news + wechat-article-search | ★★★ |
| 推理分析 | adp-prompt-builder | — | adp-prompt-builder（提示词定义交叉校验规则） | ★★★ |
| 内容生成 | 提示词内置模板 | word-docx/excel-xlsx（如需导出文件） | 提示词内置（表格直接输出） | ★★★ |
| 合规风控 | 提示词约束 | tencent-fact-check（不相关） | 提示词标注监管依据 | ★★☆ |
| 在线补取 | a-stock-data(巨潮) + news + wechat | — | 三者组合，提示词第4条定义优先级 | ★★★ |

### 模式选择

| 决策问题 | 投研助手回答 | 影响 |
|---------|------------|------|
| 需要读写文件/运行代码吗？ | 需要（下载PDF、pdftotext提取） | → 倾向 Claw |
| 需要平台级合规硬拦截吗？ | 不需要（投研场景无金融合规硬拦截需求） | → 不排除 Claw |
| 有固定执行流程吗？ | 没有（用户问题多样化，Agent自主决策） | → 排除单工作流 |
| 需要多个角色协作吗？ | 不需要（单Agent足够） | → 排除 Multi-Agent |
| 核心是严肃问答/知识服务吗？ | 是，但需要灵活调用多种Skill | → Claw 或标准 |

**最终选择：Claw 模式**

理由：
1. Skills 即装即用，13个技能灵活组合，Agent自主决定何时调哪个
2. 投研场景无合规硬拦截需求（不像财富顾问必须拦截"承诺收益"）
3. Claw 有独立工作空间，可运行代码（如 pdftotext 提取PDF）

!!! danger "Claw 模式注意事项"
    Claw 不可切换、无长期记忆、无平台级内容安全——投研场景可接受。

!!! tip "培训要点"
    对比财富顾问助手选"标准模式"（需要合规硬拦截），投研助手选"Claw模式"（需要灵活调用Skills）。**同一个平台、不同的业务场景，模式选择完全不同**。

## 第四步：设计推理

**核心问题**：为什么选它？边界与兜底在哪？

### 决策1：知识库 vs API

| 维度 | 选择 | 理由 |
|------|------|------|
| 选它 | 两者结合 | 知识库存年报/招股书原文（有出处），API补结构化数据（精确），交叉校验是亮点 |
| 不选 | 纯知识库 | 知识库覆盖有限（仅20份文档），PDF表格解析可能丢结构 |
| 不选 | 纯API | API返回的是结构化数字，没有"原文表述"，无法满足合规溯源要求 |
| 边界 | 知识库仅覆盖4家公司20份文档；API仅覆盖上市公司的财务数据 | 非上市公司、非财务指标 → 两者都查不到 |
| 兜底 | 知识库未命中 → 在线补取（巨潮搜公告/news搜新闻/wechat搜分析） | 进阶版第4条提示词定义了5条补取路径 |

### 决策2：应用模式

| 维度 | 选择 | 理由 |
|------|------|------|
| 选它 | Claw 模式 | 13个Skill灵活调用，Agent自主决策何时检索知识库、何时调API、何时补取 |
| 不选 | 标准模式 | 标准模式通过连接器调外部数据，不如Skills即装即用方便；且投研无合规硬拦截需求 |
| 不选 | Multi-Agent | 单Agent足够，不需要多角色并行 |
| 边界 | Claw无平台级内容安全/保守回答/隐私脱敏/长期记忆 | 投研场景可接受（无合规硬拦截需求） |
| 兜底 | 合规风控通过提示词软约束实现 | 标注监管依据、不承诺收益、免责声明 |

### 决策3：提示词第4条改造（基础版→进阶版的核心差异）

| 维度 | 基础版 | 进阶版 |
|------|--------|--------|
| 选它 | "再尝试调用实时数据接口补充"（模糊） | 5条明确补取路径（财务→公告→新闻→分析→标注） |
| 边界 | Agent不知道"实时数据接口"是哪个，大概率不触发 | 5条路径覆盖了所有补取场景 |
| 兜底 | 知识库未命中 → 回答"暂无"后停止 | 知识库未命中 → 按优先级补取 → 仍无结果才回答"暂无" |
| 改动量 | — | +187字符，1条规则从1句变5行 |

### 决策4：news + wechat-article-search 的价值边界

| Skill | 能做什么 | 不能做什么 | 兜底 |
|-------|---------|-----------|------|
| news | 财经新闻关键词搜索，5条/次 | 无全文，SourceUrl不可打开 | 摘要级信息，引导用户去原站看 |
| wechat | 公众号文章搜索，10条/次 | 无全文，真实URL不稳定 | 概要级信息，标注公众号名称 |
| 组合价值 | 知识库只有历史公告，news+wechat补实时动态和第三方观点 | 不能替代年报/招股书的深度提取 | 在线检索结果标注"[来源：在线检索]"，与知识库原文区分 |

### 搭建模式推荐结论

```
📋 搭建模式推荐结论

推荐模式：Claw 模式
推理依据：
  1. 13个Skill需灵活组合调用 → Skills即装即用最方便
  2. 投研场景无合规硬拦截需求 → 不需要标准模式的内容安全/保守回答
  3. 需要运行代码（pdftotext等）→ Claw有独立工作空间
  4. 无固定执行流程 → 排除单工作流
  5. 单Agent足够 → 排除Multi-Agent

替代方案：标准模式（知识库原生集成更稳定，但连接器不如Skills灵活）
⚠️ 关键风险：Claw不可切换；无平台级硬拦截；无长期记忆
🔒 合规风险：低（投研场景无金融合规硬拦截需求，提示词软约束足够）
```

!!! tip "培训要点"
    第四步是六步中最容易被跳过的一步——"选它/不选/边界/兜底"四维论证看起来啰嗦，但它逼着你思考"如果这个方案失效了怎么办"。进阶版提示词第4条的改造，正是第四步"兜底"思考的产物。

## 整体总结：六步推理链完整闭环

```
第一步 业务产出物
  → 交付物：关键指标对照表 + 关联交易清单 + 实时资讯
  ↓
第二步 能力拆解
  → P0：知识检索/数据查询/推理分析/内容生成
  → P1：合规风控/在线补取（进阶版升P0）
  ↓
第三步 功能映射 + 模式选择
  → 知识库 + a-stock-data + news + wechat + 提示词
  → Claw模式（Skills灵活，无合规硬拦截需求）
  ↓
第四步 设计推理
  → 知识库vs API：两者结合，交叉校验是亮点
  → 边界：知识库仅20份文档 → 兜底：在线补取5条路径
  → news/wechat能力边界：摘要级，不能替代原文
  ↓
第五步 分步操作
  → 基础版：11 Skill + 提示词3674字符
  → 进阶版：13 Skill + 提示词3861字符（+187字符改第4条）
  ↓
第六步 验收
  → TC-001~003：基础功能（两版都通过）
  → TC-004~006：边界与补取（基础版失败，进阶版通过）
  → 验收不过 → 回溯到第四步（兜底方案）或第三步（补Skill）
```

> **一句话总结**：六步推理链不是6个孤立的步骤，而是一条**因果链**——交付物决定能力，能力决定功能，功能决定设计，设计决定操作，操作决定验收。改任何一步，后面全部要跟着调。进阶版改了第四步的"兜底"（加在线补取路径），第五步的操作就多了2个Skill，第六步的验收就多了3个测试用例。
